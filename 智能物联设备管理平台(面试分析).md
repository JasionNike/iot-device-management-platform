## 2. 业务流程详解

### 2.1 设备注册流程

```
设备上电 → MQTT CONNECT → 发送注册消息
    │
    ▼
MqttGateway.handleMessage("iot/devices/{deviceId}/register", payload)
    │
    ▼
MqttMessageHandler.dispatch(deviceId, "register", payload)
    │
    ▼
DeviceServiceImpl.autoRegister(deviceSn, productKey, firmwareVersion)
    │
    ├─ Redis SETNX 幂等锁 ("device:register:{deviceSn}", 30s TTL)
    │   └─ 锁获取失败 → 等待已有注册完成 → 查DB返回已有设备
    │   └─ 锁获取成功 ↓
    ├─ 查 deviceSn 是否已注册 (SELECT BY device_sn)
    │   └─ 已存在 → 直接返回 (幂等)
    ├─ 生成 UUID deviceId + 16位 deviceSecret
    ├─ INSERT INTO t_device (device_id, device_sn, product_key, ...)
    ├─ 初始化空设备影子 → shadowService.getShadow(deviceId)
    │   └─ INSERT INTO t_device_shadow (空 reported/desired, version=0)
    ├─ 释放幂等锁
    └─ 返回注册成功
```

**关键设计决策**：
- 为什么用SETNX而非数据库唯一约束？SETNX提供30秒窗口期的并发保护，DB唯一约束是最终兜底。两者互补，不依赖单一机制。
- 为什么注册时初始化空影子？后续遥测上报直接update，不需要判断影子是否存在（减少分支逻辑）。

### 2.2 设备遥测 → 告警联动流程

```
DeviceSimulator/真实设备 → MQTT PUBLISH telemetry
    │
    ▼
MqttGateway.handleMessage("iot/devices/{deviceId}/telemetry", json)
    │
    ▼
MqttMessageHandler.handleTelemetry(deviceId, payload)
    │
    ├─ ① 写入遥测历史 (t_device_telemetry)
    │     INSERT INTO t_device_telemetry (device_id, property_name, property_value, data_type, report_time)
    │
    ├─ ② 更新设备影子 (reported)
    │     shadowService.updateReported(deviceId, reportedData)
    │       ├─ Redisson RLock("device:lock:{deviceId}") 获取设备级锁
    │       ├─ 读取当前影子 → 更新 state.reported + metadata.reported 时间戳
    │       ├─ CAS version+1 → saveShadowToDb() → 写Redis
    │       └─ 释放锁
    │
    ├─ ③ 发送 RabbitMQ 事件 (异步)
    │     eventProducer.sendTelemetryEvent(deviceId, data)
    │       → Exchange: iot.exchange, RoutingKey: iot.device.event
    │
    └─ ④ RabbitMQ → AlertConsumer 消费
          alertService.processEvent(deviceId, eventType, eventData)
            ├─ 查询所有启用规则 (SELECT * FROM t_alert_rule WHERE is_enabled=1)
            ├─ 遍历规则 → matchCondition(conditionJson, eventData)
            │   ├─ 解析JSON条件: {"property":"temperature","operator":">","threshold":60}
            │   ├─ 从 eventData 取出对应 property 值
            │   ├─ 按 operator 比较 (支持 >, >=, <, <=, ==, !=)
            │   └─ 匹配成功 → 下一步
            ├─ 告警收敛检查: 同 device+同 rule 在30分钟内已有告警？
            │   └─ 是 → 跳过（防告警风暴）
            │   └─ 否 ↓
            ├─ INSERT INTO t_alert_record (alertId, deviceId, ruleId, alertLevel, handleStatus=PENDING, ...)
            ├─ 异步发送通知 (Feign → Notification Stub)
            └─ 每5分钟 checkAlertEscalation(): P2+超过30分钟未处理 → 升级为P1
```

**关键设计决策**：
- 为什么遥测先写DB再发MQ？保证数据不丢失。MQ异步处理告警，遥测数据已持久化，即使MQ异常也不影响数据完整性。
- 为什么告警收敛30分钟？IoT设备可能在故障边界反复抖动（如温度59.8→60.1→59.9），30分钟窗口既避免告警风暴又不遗漏真实告警。

### 2.3 OTA升级灰度发布流程

```
管理员创建固件 → 创建OTA任务 → 启动灰度(10%)
    │
    ▼
设备定时拉取/平台推送 → 下载固件 → 安装 → 上报进度
    │
    ▼
OtaServiceImpl.updateProgress(deviceId, taskId, status, progress, failReason)
    ├─ 查 OtaRecord (deviceId + taskId)
    ├─ 存在 → UPDATE (行锁保护)
    │   └─ FAILED 且 retryCount<3 → 重置PENDING + retryCount+1
    └─ 不存在 → INSERT 新记录

运维确认批次稳定 → 手动推进灰度 +10%
    │
    ▼
OtaServiceImpl.advanceGrayPercent(taskId)
    ├─ grayPercent=10 → 20 → 30 → ... → 100
    └─ grayPercent=100 → taskStatus=COMPLETED
```

**关键设计决策**：
- 为什么手动推进而非全自动？固件升级是高风险操作，需要人工确认每批次稳定性后再推进。全自动灰度在商用IoT平台中极少使用。
- 为什么最多重试3次？经验值——3次失败大概率是固件本身问题（而非网络抖动），继续重试浪费设备带宽和电量。

### 2.4 设备影子端云协同流程

```
设备上报影子：
  Device → MQTT telemetry → handleTelemetry()
    → shadowService.updateReported(deviceId, {"temperature": 25.5})
      ✅ state.reported.temperature = 25.5
      ✅ metadata.reported.temperature.timestamp = now

平台下发期望：
  管理员 → REST API → shadowService.setDesired(deviceId, {"target_temp": 26})
    ✅ state.desired.target_temp = 26

设备拉取并执行：
  Device → REST API → shadowService.pullAndClearDesired(deviceId)
    → 返回 {"target_temp": 26}
    → 清除 state.desired = {}
    → 设备执行后上报新的 reported
```

## 3. 技术方案与实现细节

### 3.1 设备影子：三级缓存架构

```
读取路径:
  1. Redis 命中 → 直接返回 (P99 < 5ms)
  2. Redis 未命中 → 查 MySQL → 回写 Redis → 返回 (P99 < 50ms)
  3. MySQL 也不存在 → 返回空影子结构 (兜底)

写入路径:
  1. Redisson RLock (deviceId粒度, tryLock 3s)
  2. 读取当前影子 → 更新字段 → version+1
  3. 写 MySQL (乐观锁) → 写 Redis (SET 7天TTL)
  4. 释放锁
```

**为什么三级而非两级？** 返回空结构比返回null/404更好——前端和下游不需要null判断，始终拿到合法JSON。

**为什么用Redisson而非Jedis？** Redisson内置Watchdog自动续期，不需要手动设置leaseTime；Jedis需要额外实现锁续期逻辑。见[选型分析 4.2](#42-分布式锁选型)。

### 3.2 告警收敛：时间窗口 + 防抖

```java
// 30分钟窗口内同设备+同规则不重复创建
String convergeKey = deviceId + ":" + rule.getId();
if (System.currentTimeMillis() - lastAlertTime < 30 * 60 * 1000) {
    return; // 跳过
}
```

**为什么不直接用Redis去重？** 去重逻辑在Service层用时间戳比较更直观，避免额外Redis调用。如果设备量达到百万级，可以用Redis的`SET key value EX 1800 NX`实现更高效的分布式去重。

### 3.3 通知补发：DLX + 指数退避

```
第一次通知失败
  → 写入 DB (t_notification_retry, status=PENDING, retryCount=0)
  → 发送到 wait.queue (TTL=1min)
      │
      ▼ (1分钟后TTL过期)
  → dead-letter → retry.exchange → retry.queue
  → NotificationRetryConsumer 消费
      ├─ 成功 → ACK + UPDATE DB status=SENT
      └─ 失败 → retryCount++ → 计算下次延迟 = min(30min, 2^retryCount min)
                → 重新发送到 wait.queue

重试次数 >= 5 → 进入 DLQ → DB 标记 FAILED → 人工介入
```

**为什么用指数退避而非固定间隔？** 固定间隔可能持续重试失败的服务（如对方宕机30分钟），指数退避让开服务恢复时间窗口，减少无效重试。

### 3.4 多厂商协议适配：策略 + 工厂模式

// 工厂自动注册

    // 扫描所有 ProtocolAdapter Bean

    // adapterMap.put("SENSOR-TH-001", sensorAdapter);

### 3.5 JWT认证 + 登录限流

```
登录流程:
  1. IP限流检查 (Redis "login:lock:{ip}" )
  2. 验证码校验 (Redis "captcha:{captchaKey}", 5分钟TTL, 一次性)
  3. BCrypt密码校验
  4. 签发JWT (HMAC-SHA256, sub/roles/iat/exp, 2小时)
  5. 删除验证码 + 清除失败计数

请求认证:
  Gateway AuthGlobalFilter → 提取 Bearer Token → 透传
  Backend JwtAuthenticationFilter → 校验 → SecurityContext
```

## 4. 技术选型分析

### 4.1 微服务框架：Spring Cloud Alibaba vs Spring Cloud Netflix vs Dubbo

| 维度 | SCA (选用) | Spring Cloud Netflix | Dubbo |
|------|-----------|---------------------|-------|
| 服务注册 | Nacos (注册+配置一体) | Eureka (仅注册，已停维护) | Zookeeper (需额外部署) |
| 熔断限流 | Sentinel (Dashboard可视化) | Hystrix (已停更) | 需集成Sentinel |
| 配置中心 | Nacos (热刷新) | Spring Cloud Config | 需额外方案 |
| 网关 | Spring Cloud Gateway (响应式) | Zuul 1.x (阻塞式，已停更) | 无内置 |
| 社区活跃度 | 阿里持续维护 | Netflix组件陆续停更 | Apache维护，更新慢 |
| 学习成本 | 与Spring Boot深度整合 | 同上 | 独立生态 |

**结论**：Spring Cloud Alibaba是2025年国内商用IoT项目的事实标准。Netflix技术栈已停止维护（Eureka/Hystrix/Zuul均已停更），Dubbo更适合纯RPC场景，缺少网关/配置中心一体方案。

### 4.2 分布式锁：Redisson vs Jedis vs Spring Integration Lock

| 维度 | Redisson (选用) | Jedis | Spring Integration |
|------|----------------|-------|--------------------|
| Watchdog自动续期 | ✅ 内置，默认10s续期 | ❌ 需手动实现 | ❌ 需手动实现 |
| 锁粒度 | RLock (可重入) | set nx px + lua脚本 | RedisLockRegistry |
| 读写锁/信号量 | ✅ 支持 | ❌ 需自己实现 | ❌ 不支持 |
| 与Spring整合 | ✅ redisson-spring-boot-starter | ❌ 需封装 | ✅ 原生支持 |
| 社区活跃 | 21k+ Star | 12k+ Star | Spring体系 |

**为什么不用Jedis实现锁？** Jedis的SET NX PX需要开发者手动处理锁续期、锁释放的原子性（Lua脚本）、RedLock算法。Redisson封装了这些细节，例如`lock.tryLock(3, SECONDS)`时如果指定leaseTime则不续期，不指定则Watchdog每10秒自动续期——开发只需关注业务逻辑。

**实际代码中的体现**：
```java
// ShadowServiceImpl - 影子并发控制
RLock lock = redissonClient.getLock("device:lock:" + deviceId);
if (!lock.tryLock(3, TimeUnit.SECONDS)) {
    throw new BusinessException("获取分布式锁失败");
}
try {
    // 业务逻辑...（Watchdog自动续期，无需手动处理锁超时）
} finally {
    lock.unlock();
}
```

### 4.3 消息队列：RabbitMQ vs Kafka vs RocketMQ vs Pulsar

| 维度 | RabbitMQ (选用) | Kafka | RocketMQ | Pulsar |
|------|----------------|-------|----------|--------|
| 消息模型 | 丰富(Exchange/Queue/RoutingKey) | 简单(Partition) | 丰富(类似RabbitMQ) | 丰富 |
| 延迟消息 | DLX+TTL实现 | 需额外处理 | ✅ 原生支持(18级) | ✅ 原生支持 |
| 运维复杂度 | 低(镜像集群) | 高(ZK依赖) | 中(NameServer) | 高(BookKeeper) |
| 适用场景 | 业务事件驱动、任务分发 | 日志/埋点/大数据管道 | 电商交易、金融 | 多租户 |
| 资源消耗 | 低(Erlang, 百MB级) | 高(JVM, GB级) | 中 | 高 |
| 消息确认 | 手动ACK, 精确控制 | 偏移量提交 | 手动/自动 | 手动/自动 |
| 社区成熟度 | ✅ 成熟(15年+) | ✅ 成熟 | ⚠️ 国内为主 | ⚠️ 较新 |

**为什么选RabbitMQ而非Kafka？** IoT平台的设备事件是**业务驱动型**消息（告警、通知、状态变更），特点是消息量中等但路由规则复杂、需要延迟重试。RabbitMQ的Exchange/Queue/RoutingKey模型完美匹配"不同类型事件路由到不同消费者"的需求。Kafka擅长海量日志流式处理，对于IoT的设备事件路由反而过于重量级。

**为什么没用RocketMQ的原生延迟消息？** 项目需要指数退避(1min→2min→4min→...→30min)，而RocketMQ只支持18个固定延迟级别。RabbitMQ的Per-Message TTL + DLX方案支持任意延迟时间，更灵活。

### 4.4 数据库版本迁移：Flyway vs Liquibase

| 维度 | Flyway (选用) | Liquibase |
|------|--------------|-----------|
| 学习成本 | 低(SQL就是迁移) | 高(需学XML/YAML/JSON DSL) |
| 版本管理 | 文件名约定(V1__xxx.sql) | CHANGELOG表 |
| 回滚支持 | ❌ 社区版不支持 | ✅ 支持 |
| Spring整合 | ✅ flyway-core | ✅ liquibase-core |
| 适合场景 | SQL为主的项目 | 跨数据库类型项目 |

**为什么选Flyway？** IoT项目的变更全部是MySQL DDL，SQL版本化最直接。Liquibase的多数据库支持在本项目不需要；Flyway的简单性更符合团队快速上手的需求。

### 4.6 加密：Jasypt vs Spring Cloud Config加密 vs Vault

| 维度 | Jasypt (选用) | Spring Cloud Config | HashiCorp Vault |

**选用考量**：代码在GitHub，GitHub Actions零额外部署。Jenkins需要额外服务器（ECS已资源紧张），GitLab CI不适用。

## 5. 面试高频追问准备

### Q1: 设备影子为什么不用Redis Stream或直接Canal同步？

**答**：设备影子是**设备级别的实时读写**场景，不是流式同步场景。Redis作为缓存提供读写分离——读走Redis(P99<5ms)，写同时更新Redis+MySQL。如果设备量达到百万级且需要影子变更历史，可引入Canal监听MySQL binlog实现影子变更事件的异步推送。

### Q2: 告警规则只有6条？生产环境怎么扩展？

**答**：当前6条是演示环境的预置规则。生产环境方案：
- 规则存储在DB（已支持），管理员通过UI增删改
- 引入规则引擎（如Drools/EasyRules），将条件匹配从硬编码if-else升级为DSL规则表达式
- 规则变更后动态刷新（Nacos配置中心热刷新规则列表）
- 复杂规则（如连续3次超阈值才告警）用CEP（复杂事件处理）引擎处理

### Q3: 设备量到百万级，当前架构哪里会先出问题？

**答**：
1. MySQL：单表2000万行后查询性能下降 → 按device_id分表（ShardingSphere）或按时序分表（按月）
2. Redis：内存限制 → 冷设备影子LRU淘汰，热点设备影子始终缓存
3. RabbitMQ：消息积压 → 增加消费者实例 + 设置消息TTL + 监控DLQ深度
4. MQTT连接：百万长连接 → 引入EMQX集群做MQTT Broker，网关服务只处理业务逻辑

### Q4: OTA灰度失败怎么回滚？

**答**：当前代码的灰度推进是手动确认的——运维在Dashboard看升级成功率，确认正常后才点"+10%"。如果批次异常：
1. 停止灰度推进（不再+10%）
2. 创建回滚OTA任务（target_firmware=上一个稳定版本）
3. 已升级的少量设备通过新任务回滚
4. 升级记录完整保留（t_ota_record），事后复盘

### Q5: 设备注册的UUID deviceId在高并发下会有冲突吗？

**答**：UUID v4的碰撞概率为2^122分之一（约10^-37量级），比硬件故障概率还低。即使极端情况下碰撞，数据库的`UNIQUE KEY uk_device_id`是最终兜底。实际生产中从未遇到过UUID碰撞案例。

### Q6: 为什么不直接用MQTT协议的遗嘱消息(Will Message)来检测设备离线？

**答**：MQTT遗嘱消息存在局限——需要MQTT Broker和平台始终保持连接、遗嘱消息可能因Broker重启丢失。当前方案使用心跳超时（missCount>=3）判断离线，不依赖MQTT Broker状态，更健壮。两者可在生产环境互补使用：心跳为主、遗嘱为辅。

### Q7: SecurityConfig中为什么禁用了CSRF？

**答**：CSRF攻击的前提是Cookie-based Session认证。项目使用JWT无状态认证（Token在Header中），浏览器不会自动携带，因此天然免疫CSRF。禁用CSRF是正确做法，启用反而会阻止正常的POST/PUT请求。

### Q8: 项目中用到了策略模式，还有什么设计模式？

**答**：
- **策略模式**：ProtocolAdapter（协议解析）
- **工厂模式**：ProtocolAdapterFactory（按productKey路由适配器）
- **代理模式**：Spring AOP（AuditLogAspect审计日志）
- **观察者模式**：RabbitMQ发布订阅（EventProducer → AlertConsumer）
- **模板方法模式**：AbstractProtocolAdapter（定义标准流程，子类扩展）
- **单例模式**：Spring Bean默认单例
- **适配器模式**：各个ProtocolAdapter将厂商私有协议适配为标准数据格式
