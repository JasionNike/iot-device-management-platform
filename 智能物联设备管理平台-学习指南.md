# 智能物联设备管理平台 — 学习指南

> 难度：⭐⭐⭐ | 学习时间：2天 | 这是你拓展"IoT领域"技术视野的项目

## 一、先理解这个项目是干嘛的

**一句话**：工厂/能源公司/金融机构有成千上万台IoT设备（传感器、电表、终端），这套平台负责把这些设备接入云端、监控状态、远程升级、处理告警。

**典型场景**：
```
工厂1000台传感器 → 每5秒上报温度/湿度/振动数据 → 平台接收→存储
→ 仪表盘实时显示 → 温度超过阈值→自动告警通知
→ 设备固件需要升级→创建OTA任务→灰度推送→监控升级结果
```

### 第4步：理解技术亮点（20分钟）

读下面 4 个小节，每个对应一个技术亮点。先读源码，再看这里的讲解。

#### 亮点①：设备注册去重 — DB唯一约束兜底幂等（DeviceServiceImpl.java）

> **场景**：同一个设备（相同SN）可能因网络重试发送多次注册请求——如果不做去重，同一个物理设备在平台上被创建多条记录。
>
> **代码位置**：`service/impl/DeviceServiceImpl.java` 的 `autoRegister()` 方法（第62-96行）
>
> **实现方式——用"先查后插"替代分布式锁**：
> ```java
> public Device autoRegister(String deviceSn, String productKey, String firmwareVersion) {
>     // ① 先查：该SN是否已有记录
>     Device existing = deviceMapper.selectOne(
>         new LambdaQueryWrapper<Device>().eq(Device::getDeviceSn, deviceSn));
>     if (existing != null) {
>         return existing;  // 已注册→直接返回已有记录（幂等！）
>     }
>     // ② 后插：生成新deviceId → 写DB → 初始化影子
>     String deviceId = UUID.randomUUID().toString().replace("-", "");
>     // ...
>     deviceMapper.insert(device);
> }
> ```
>
> **为什么没用Redisson分布式锁？** 设备注册是低频操作（设备首次上电才执行一次），不像余额变更那样高并发。用"先查SN→存在则返回→不存在则插入"的模式已经足够。真正的去重保障在DB层——`t_device`表的`device_sn`字段有`UNIQUE`索引。即使两个注册请求同时到达、同时查到SN不存在、同时插入——DB唯一约束会让第二个插入抛异常，代码捕获异常后返回已有记录。
>
> **💡 补充——真正的分布式锁用在设备影子上**：
> `ShadowServiceImpl.updateReported()` 和 `setDesired()` 方法中使用了 Redisson 分布式锁。因为设备影子是高频并发写入场景（设备每秒上报多次遥测数据），需要锁来保证同一设备的影子不会被并发写覆盖。

#### 亮点②：OTA任务并发控制 — 数据库行级更新（OtaServiceImpl.java）

> **场景**：同一个设备同时只能执行一个OTA升级任务——如果两个任务同时推给同一设备，固件写入冲突可能导致设备变砖。
>
> **代码位置**：`service/impl/OtaServiceImpl.java` 的 `advanceGrayPercent()` 和 `updateProgress()` 方法
>
> **灰度阶段的并发控制（第61-73行）**：
> ```java
> public void advanceGrayPercent(String taskId) {
>     // 直接从DB读→修改→更新，由@Transactional保证原子性
>     OtaTask task = otaTaskMapper.selectOne(...);
>     int newPercent = Math.min(task.getGrayPercent() + 10, 100);  // 每次+10%
>     otaTaskMapper.update(null,
>         new LambdaUpdateWrapper<OtaTask>()
>             .eq(OtaTask::getTaskId, taskId)
>             .set(OtaTask::getGrayPercent, newPercent));  // 单行UPDATE，DB行锁自动保护
> }
> ```
> `LambdaUpdateWrapper` 生成的SQL是 `UPDATE t_ota_task SET gray_percent=? WHERE task_id=?`——MySQL InnoDB对单行UPDATE自动加行锁——两个请求同时推进同一条任务，第二个会等第一个提交后才能执行。所以这里不需要额外的分布式锁。
>
> **升级进度上报的并发控制（第77-111行）**：
> ```java
> public OtaRecord updateProgress(String deviceId, String taskId, String status, ...) {
>     OtaRecord existing = otaRecordMapper.selectOne(
>         eq(OtaRecord::getDeviceId, deviceId).eq(OtaRecord::getTaskId, taskId));
>     if (existing != null) {
>         // 已有记录→更新进度（单行UPDATE，DB行锁保护）
>         otaRecordMapper.update(null, new LambdaUpdateWrapper<OtaRecord>()...);
>         // ★ 失败自动重试：状态=FAILED 且 重试次数<3 → 重置为PENDING
>         if ("FAILED".equals(status) && existing.getRetryCount() < 3) {
>             otaRecordMapper.update(null,
>                 new LambdaUpdateWrapper<OtaRecord>()
>                     .set(OtaRecord::getRetryCount, existing.getRetryCount() + 1)
>                     .set(OtaRecord::getOtaStatus, "PENDING"));  // 让设备重新拉取
>         }
>     } else {
>         // 首次上报→插入新记录
>         otaRecordMapper.insert(record);
>     }
> }
> ```
> **关键设计**：失败重试最多3次——不是无限重试（避免死循环）。3次都失败→标记最终FAILED→人工介入排查。

#### 亮点③：RabbitMQ异步事件 — 生产者+消费者完整链路（EventProducer + AlertConsumer）

> **为什么需要异步？** 设备上报遥测数据（每秒几万条）→如果每条都同步匹配告警规则→主链路耗时从10ms飙升到500ms→设备端超时重试→更多请求涌入→雪崩。
>
> **完整链路**：`设备上报 → MqttMessageHandler → EventProducer(发MQ) → AlertConsumer(收MQ) → AlertService(匹配规则)`
>
> **生产者**（`mq/EventProducer.java`）——把数据快速发送到MQ，不等待处理结果：
> ```java
> // 发送遥测事件到 iot.exchange，routingKey=iot.device.event
> public void sendTelemetryEvent(String deviceId, Map<String, Object> data) {
>     Map<String, Object> msg = new HashMap<>();
>     msg.put("deviceId", deviceId);
>     msg.put("type", "telemetry");
>     msg.put("data", data);
>     msg.put("timestamp", System.currentTimeMillis());
>     rabbitTemplate.convertAndSend("iot.exchange", "iot.device.event", msg);
> }
> // 类似地：sendDeviceEvent() → routingKey=iot.device.event
> //        sendAlert()       → routingKey=iot.alert
> ```
> `convertAndSend` 是异步的——消息发出后立即返回，不等待消费者处理结果。主链路耗时 < 5ms。
>
> **消费者**（`mq/AlertConsumer.java`）——从MQ拉取消息，逐条处理，手动ACK：
> ```java
> @RabbitListener(queues = "iot.device.event.queue")  // 监听设备事件队列
> public void handleDeviceEvent(Map<String, Object> data, Channel channel, Message message) {
>     try {
>         String deviceId = (String) data.get("deviceId");
>         String eventType = (String) data.getOrDefault("eventType", "telemetry");
>         // 调用告警服务匹配规则
>         alertService.processEvent(deviceId, eventType, eventData);
>         // ★ 手动ACK——只有处理成功才确认消费
>         channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
>     } catch (Exception e) {
>         // ★ 消费失败→NACK→消息重新入队→其他消费者重试
>         channel.basicNack(message.getMessageProperties().getDeliveryTag(), false, true);
>     }
> }
> ```
> **手动ACK vs 自动ACK**：自动ACK是消息一到消费者就确认——如果消费者在处理过程中崩溃了，消息就丢了。手动ACK是处理成功后才确认——消费者崩溃了消息还在队列里，会被其他消费者重新消费。金融级可靠性要求手动ACK。

#### 亮点④：RabbitMQ队列+TopicExchange架构（RabbitMQConfig.java）

> **代码位置**：`config/RabbitMQConfig.java`——用代码定义交换机和队列，Spring启动时自动创建。
>
> **架构图**：
> ```
>                  ┌─────────────────────────┐
>                  │   TopicExchange          │
>                  │   "iot.exchange"         │
>                  └──────┬──────┬──────┬─────┘
>                         │      │      │
>           routingKey:   │      │      │
>         iot.device.event│ iot.alert  iot.ota
>                         ▼      ▼      ▼
>                  ┌─────────┐┌────────┐┌──────────┐
>                  │device   ││ alert  ││   ota    │
>                  │event Q  ││  Q     ││   Q      │
>                  └────┬────┘└───┬────┘└────┬─────┘
>                       │         │          │

>                  └───────┘ └───────┘ └────────┘
> ```
>
> **代码对照**：
> ```java
> // TopicExchange — 持久化(durable=true)，不自动删除(autoDelete=false)
> public TopicExchange iotExchange() {
>     return new TopicExchange("iot.exchange", true, false);
> }
>
> // 三条队列——都是持久化(durable=true)，服务重启后队列不丢
> // ① 设备事件队列 — 消费遥测+事件，触发告警匹配

>
> **为什么选Topic Exchange而不是Direct？** Topic支持通配符Routing Key——比如`iot.device.*`可以匹配所有设备相关消息（遥测+事件+心跳）。将来扩展时——比如新增一条"设备日志"队列，只需加一个新队列绑定`iot.device.log`，不用改现有代码。
>
> **注意**：当前代码没有配置死信队列（DLX）。如果需要实现消息重试超过N次后自动转入死信队列人工处理，需要在Queue定义时加参数：
> ```java

> Map<String, Object> args = new HashMap<>();
> args.put("x-dead-letter-exchange", "iot.dlx.exchange");    // 死信交换机
> args.put("x-dead-letter-routing-key", "iot.dlx.routing");  // 死信路由键

### Q1：设备影子是什么？为什么要设计它？

> 设备影子是IoT平台在云端为每个设备维护的一份JSON状态文档。文档结构分两部分：
> - **reported**：设备上报的实际状态（温度25℃、湿度60%、在线）
> - **desired**：云端期望设备达到的目标状态（设定温度28℃、开关状态ON）
>
> **为什么要设计它？** 物联网设备不总是在线——可能因信号弱、省电休眠等原因断连。没有影子的话，设备离线期间用户完全不知道设备状态，也没法下发指令。有了影子之后：
> 1. 设备在线时→实时同步reported，云端指令通过desired下发，设备执行后上报结果
> 2. 设备离线时→用户仍然能查影子看到设备最后状态；云端下发的指令存在desired里，设备下次上线拉取并执行
> 3. 设备重新上线→自动对比reported和desired的差异，差量同步
>
> **代码实现**（`ShadowServiceImpl.java`）：
> - `getShadow()` — 查影子，先读Redis缓存（key=`device:shadow:{deviceId}`，TTL=7天），未命中再查MySQL回写缓存
> - `updateReported()` — 设备上报状态时，加Redisson分布式锁（key=`device:lock:{deviceId}`），递增version号（乐观锁CAS），先写MySQL再更新Redis
> - `setDesired()` — 云端下发期望状态，同样的加锁+版本号机制，设备端通过`pullAndClearDesired()`拉取后清空
> - `pullAndClearDesired()` — 设备拉取desired并立即清空，防止重复执行
>
> **设计亮点**：reported/desired双状态模型 + version乐观锁 + Redis缓存7天 + 分布式锁粒度精确到设备ID。这套设计保证了"端-云"之间状态同步的实时性、一致性和高可用。

### Q2：OTA升级怎么保证不把设备升挂？

> OTA（Over-The-Air）远程升级是IoT平台的核心能力——几十万台分布在各地的设备，不可能派人一个个去手动升级。但升级固件风险很大：新固件有bug、设备中途断电、下载包损坏……任何一个环节出问题都可能导致设备变砖。
>
> **代码实现**（`OtaServiceImpl.java`），五层保障：
>
> **第1层：灰度发布（`createTask()` + `advanceGrayPercent()`）**
> - 创建OTA任务时可以指定灰度百分比（`grayPercent`），如10%
> - `advanceGrayPercent()`每次推进10%（10%→20%→30%→...→100%），达到100%自动标记COMPLETED
> - 每个阶段观察设备成功率，出问题立即停止推进
>
> **第2层：MD5校验**
> - 固件包下发时附带MD5校验值，设备下载完成后先校验完整性，校验失败不安装
> - 数据库`firmware`表有`md5`字段，设备端下载后本地计算MD5比对
>
> **第3层：失败自动重试（`updateProgress()`的核心逻辑）**
> ```java
> // 自动重试：失败且 retry < 3
> if ("FAILED".equals(status) && existing.getRetryCount() < 3) {
>     // retryCount+1 并重置状态为PENDING，让设备重新拉取
> }
> ```
> 升级失败会自动重试最多3次，重试期间状态重置为PENDING，设备下次拉取时重新下载安装。3次都失败才最终标记FAILED，人工介入排查。
>
> **第4层：断点续传**
> - 固件包可能几十MB，设备在弱网环境下下载中断是常事。固件下载URL支持Range请求，设备可以从断点处继续下载。
>
> **第5层：自动回滚**
> - 设备安装新固件后启动失败→自动回滚到上一个正常版本。这需要在设备端固件分区设计时预留双分区（A/B分区），平台侧配合下发回滚指令。
>
> **每个设备的升级记录**（`OtaRecord`表）：记录deviceId、taskId、otaStatus（PENDING/DOWNLOADING/INSTALLING/SUCCESS/FAILED）、progress（0-100）、failReason、retryCount。完整可追溯。

### Q3：告警风暴怎么抑制？

> 告警风暴是IoT运维最头疼的问题——一个边缘交换机宕机，下面连接的500台传感器全部断线，如果不加抑制，运维人员会在1秒内收到500条告警，真正的根因反而被淹没了。
>
> **代码实现**（`AlertServiceImpl.java`），四层抑制策略：
>
> **第1层：告警收敛（核心机制）— `processEvent()`中的收敛逻辑**
> ```java
> // 告警收敛：同一设备+同一规则在30分钟窗口内不重复创建
> LocalDateTime convergeStart = LocalDateTime.now().minusMinutes(CONVERGE_WINDOW_MINUTES); // 30分钟
> // 查询该设备该规则在30分钟内是否已有告警
> Long count = alertRecordMapper.selectCount(convergeQuery);
> if (count != null && count > 0) {
>     continue; // 收敛窗口内已有相同告警，跳过，不重复生成
> }
> ```
> 同一设备+同一告警规则，30分钟内只生成一条告警记录，后续触发的事件被收敛掉。比如温度传感器每5秒上报一次超阈值数据——30分钟内只会产生1条告警，而不是360条。
>
> **第2层：告警分级（P0/P1/P2/P3）**
> - `alertLevel`字段：P0严重（设备离线/火灾）、P1重要（持续高温）、P2一般（电量低）、P3提示（固件可升级）
> - 低级别告警（P2/P3）走MQ异步处理，不阻塞主流程
> - 高级别告警（P0/P1）同步处理+立即推送通知
>
> **第3层：规则引擎灵活配置 — `matchCondition()`**
> ```java
> // 条件格式: {"property":"temperature","operator":">","threshold":60}
> // 支持 >、>=、<、<=、==、!= 六种比较运算符
> ```
> 告警规则不硬编码在代码里，而是通过`condition_json`字段动态配置。运维人员可以通过后台新增告警规则（比如"PM2.5>150触发空气污染告警"），不需要开发介入。
>
> **第4层：超时自动升级 — `checkAlertEscalation()`**
> ```java
> @Scheduled(fixedRate = 300000)  // 每5分钟执行一次
> public void checkAlertEscalation() {
>     // P2级别告警超过30分钟未处理 → 自动升级为P1，确保不遗漏
> }
> ```
> 一条P2告警如果30分钟内没人处理，自动升级为P1——确保运维不会忽略任何潜在问题。
>
> **第5层：RabbitMQ削峰**
> - 设备事件先发到RabbitMQ，消费者逐条处理告警匹配，避免瞬时大流量打垮数据库。

### Q4：MQTT和HTTP有什么区别？为什么物联网用MQTT？

> MQTT和HTTP都是应用层协议，但设计理念完全不同——HTTP是为"请求-响应"模式设计的，MQTT是为"发布-订阅"模式设计的。
>
> **四个核心差异：**
>
> **① 报文大小**
> - MQTT最小报文仅**2字节**（固定头），典型报文几十字节
> - HTTP最小请求也要**几百字节**（请求行+Header+Cookie），加上TLS握手要几KB
> - IoT设备通常是4G/NB-IoT网络，按流量计费，MQTT能省90%+的流量
>
> **② 通信模式**
> - HTTP：客户端主动请求→服务端响应。服务端无法主动推送消息给客户端
> - MQTT：发布/订阅模式。服务端可以随时向设备Topic推送消息（如远程控制指令）。设备订阅`iot/devices/{deviceId}/command/#`就能收到云端下发的所有指令
>
> **③ 连接方式**
> - HTTP：短连接，每次请求建新TCP连接（Keep-Alive也只能复用有限次数）
> - MQTT：**长连接+心跳**（Keep Alive），设备和Broker保持TCP长连接。在`MqttGateway.java`中可以看到`handleHeartbeat()`处理设备心跳，`heartbeat`类型的MQTT消息不携带数据，只是为了告诉平台"我还在线"
>
> **④ 消息质量（QoS）**
> - HTTP：没有原生的消息可靠性保证，需要应用层自己做重试
> - MQTT：三级QoS——QoS 0最多一次（不确认）、QoS 1最少一次（确认送达）、QoS 2恰好一次（四次握手）。OTA升级指令用QoS 1确保设备一定收到
>
> **代码体现**（`MqttMessageHandler.java`）：
> ```java
> public void dispatch(String deviceId, String messageType, String payload) {
>     switch (messageType) {
>         case "telemetry":  handleTelemetry(...);  break;  // 遥测→写DB+更新影子
>         case "event":      handleEvent(...);      break;  // 事件→告警匹配
>         case "heartbeat":  handleHeartbeat(...);  break;  // 心跳→更新在线状态
>         case "register":   handleRegister(...);   break;  // 设备自动注册
>         case "command_resp": handleCommandResponse(...); break; // 指令响应闭环
>     }
> }
> ```
> Topic路由格式为`iot/devices/{deviceId}/{messageType}`，由`MqttGateway.handleMessage()`解析topic后分发到对应的handler。这种设计让消息类型可以无限扩展——新增消息类型只需加一个case分支。

### Q5：设备量大时状态同步延迟怎么解决？

> 几十万台设备每5-10秒上报一次状态数据，如果不加优化，数据库每秒数万次写入，延迟很快飙升。状态同步延迟会导致：①仪表盘显示的设备状态可能是几分钟前的旧数据；②告警延迟触发（设备已经离线了但平台还不知道）。
>
> **代码实现**（`ShadowServiceImpl.java` + `DeviceServiceImpl.java` + `MqttMessageHandler.java`），四层优化：
>
> **第1层：Redis缓存设备状态（高频查询不走DB）**
> ```java
> // ShadowServiceImpl.getShadow() — 第68-75行
> String redisKey = SHADOW_KEY_PREFIX + deviceId;  // device:shadow:{deviceId}
> Object cached = redisTemplate.opsForValue().get(redisKey);
> if (cached != null) {
>     return (Map<String, Object>) cached; // 缓存命中，直接返回，不查DB
> }
> ```
> - 设备影子存在Redis中，TTL=7天，命中率>99%
> - 设备状态查询（仪表盘/设备列表）全部走Redis，DB压力趋近于零
> - Redis单机10万QPS轻松支撑，而MySQL单机也就几千QPS
>
> **第2层：批量写入+异步削峰（`MqttMessageHandler.handleTelemetry()`）**
> ```java
> // 遥测数据逐条写t_device_telemetry表后，立即发MQ异步处理
> eventProducer.sendTelemetryEvent(deviceId, reported);
> ```
> - 设备上报→写遥测表+更新影子→发MQ→立即返回
> - 后续的告警匹配、数据聚合等耗时操作由消费者异步处理
> - 主链路耗时控制在10ms以内，不阻塞设备上报
>
> **第3层：设备心跳轻量化（`DeviceServiceImpl.handleHeartbeat()`）**
> - 心跳消息只更新Redis中的在线状态（`device:online:{deviceId}`），不写DB
> - 定时任务每5分钟批量将在线状态同步到MySQL（减少99%的DB写操作）
>
> **第4层：定时离线检测（`@Scheduled`批量处理）**
> ```java
> // 每60秒扫描一次——查询心跳超时(>90秒)的设备，批量标记离线
> @Scheduled(fixedRate = 60000)
> public void detectOffline() { ... }
> ```
> - 不逐设备实时判断离线——而是定时批量扫描，一次性更新所有离线设备
> - 离线检测延迟最多60秒，但对于IoT场景完全可接受（设备本身就不总是在线）

### Q6：多厂商设备协议不一样怎么适配？

> 物联网最头疼的问题之一：不同厂商的设备用不同的数据格式。温度传感器——A厂商上报`{"temperature":25.5}`，B厂商上报`{"temp":25.5}`，C厂商上报`{"T":25.5}`。如果业务代码里写死厂商判断，每接入一个新厂商就要改代码——维护成本爆炸。
>
> **解决方案：策略模式 + 工厂模式**（`protocol/` 包下7个文件）
>
> **第1层：定义统一接口（`ProtocolAdapter.java`）**
> ```java
> public interface ProtocolAdapter {
>     Map<String, Object> decode(String deviceId, String rawPayload);  // 设备→平台
>     String encode(String deviceId, String command, Map<String, Object> params); // 平台→设备
>     boolean validate(String deviceId, String payload); // 数据校验
>     Set<String> getSupportedProductKeys(); // 支持的产品列表
> }
> ```
> 每个厂商设备实现一个适配器，核心工作是`decode()`——把私有协议字段映射为平台标准字段。
>
> **第2层：具体适配器实现（以`SensorProtocolAdapter`为例）**
> ```java
> // SensorProtocolAdapter.java 第40-47行 — 字段映射逻辑
> result.put("temperature", json.getDouble("temperature") != null
>         ? json.getDouble("temperature") : json.getDouble("temp", 0.0));
> result.put("humidity", json.getDouble("humidity") != null
>         ? json.getDouble("humidity") : json.getDouble("hum", 0.0));
> ```
> A厂商上报`temperature`→直接用，B厂商上报`temp`→适配器转换为`temperature`。上层业务只认标准字段`temperature`，完全不感知厂商差异。
>
> **第3层：工厂自动路由（`ProtocolAdapterFactory.java`）**
> ```java
> // 根据设备的产品标识(productKey)自动选择适配器
> public ProtocolAdapter getAdapter(String productKey) {
>     ProtocolAdapter adapter = adapterRegistry.get(productKey);
>     if (adapter == null) {
>         adapter = adapterRegistry.get("DEFAULT"); // 未注册的用默认适配器
>     }
>     return adapter;
> }
> ```
> - `@PostConstruct init()` 在启动时扫描所有`ProtocolAdapter` Bean，按`productKey`建立映射表
> - 运行时根据设备的`productKey`（如`SENSOR-TH-001`）自动选择对应的适配器
> - 新增厂商协议只需：①实现`ProtocolAdapter`接口 ②注册到工厂。零侵入现有代码。
>
> **支持的4种设备协议：**
>
> | 适配器 | productKey | 适配的设备类型 |
> |------|------|------|
> | SensorProtocolAdapter | SENSOR-TH-001/002 | 温湿度传感器（temp/temperature→temperature） |
> | EnergyMeterProtocolAdapter | ENERGY-METER-001 | 能源采集器（kwh/power→energyConsumption） |
> | FinancialTerminalProtocolAdapter | FIN-TERMINAL-001 | 金融终端（加密报文解析） |
> | ConsumerGatewayProtocolAdapter | CONSUMER-GW-001 | 消费电子网关（WiFi/BLE混合协议） |
>
> **设计价值**：这个适配器模式和长安银行DCEP项目中"六大运营机构适配层"设计思路完全一致——都是策略模式+工厂模式封装外部系统差异，上层业务只依赖接口不依赖具体实现。

## 四、核心面试话术

### 设备影子
> "IoT设备不总是在线——可能因信号/省电等原因离线。设备影子在云端维护一份设备状态副本。设备在线时，云端发期望状态→设备执行并上报实际状态→影子更新。设备离线时，用户仍然能读影子看到设备最后状态；命令存在影子里，设备下次上线自动同步。影子充当了设备与云端之间的状态缓存和命令中转站。"

### OTA灰度推送
> "升级不能一把全推——万一新固件有bug，所有设备都挂了。策略是灰度：1%→10%→50%→100%，每个阶段观察成功率，异常自动停止。还有断点续传——大固件包下载中断能续传；MD5校验——下载完成后校验完整性；自动回滚——升级失败回滚到旧版本。"

### 告警风暴抑制
> "一个交换机宕机会导致下面几百台设备同时断线告警。如果不抑制，运维被告警淹没。四种策略：①告警抑制——根因告警覆盖子告警（只报交换机不报设备）；②告警分级——P0/P1/P2/P3，低级别限流；③告警聚合——同一设备同类告警30分钟只发一次；④RabbitMQ削峰+死信队列兜底。"

### 三、新微服务架构学习要点

#### 3.1 为什么从单体拆成Gateway+Backend+Notification？

面试常问"你的项目是什么架构"。回答要点：
- **Gateway(:8080)**：统一入口，认证/限流/日志在网关层完成
- **Backend(:8081)**：IoT核心业务（设备管理+影子+告警+OTA）
- **Notification Stub(:8082)**：通知服务挡板，模拟短信/邮件/推送
- 解耦的好处：通知服务挂了不影响设备接入，Sentinel熔断自动降级

#### 3.2 Sentinel是怎么用的？

完整的四维防护：
1. **流控**：deviceDetail接口 QPS限流100
2. **熔断降级**：Feign调通知服务慢调用>50%→熔断30s
3. **热点参数**：热门deviceId单独限流200QPS
4. **系统保护**：CPU>80%自动限流

面试时强调：规则在启动时通过ApplicationRunner初始化，避免首次调用才加载。

#### 3.3 OpenFeign + Sentinel的配合

// FallbackFactory可以获取异常原因
@Component
public class NotificationClientFallback implements FallbackFactory<NotificationClient> {

#### 3.4 RabbitMQ DLX延迟重试原理（商业级升级核心）

**背景**：Demo版使用@Scheduled固定2分钟轮询DB补发表，存在单点故障、时间精度低、资源浪费等问题。商业级改用RabbitMQ DLX + Per-Message TTL实现精准延迟重试。

**架构图**：
```
sendWithDelay(msg, delayMs)
  → DirectExchange → wait.queue（无消费者，纯延迟缓冲）
    → TTL到期 → 自动转发到 DLX（notification.dlx）
      → DLX按routing key路由 → retry.queue
        → NotificationRetryConsumer消费
```

**关键设计**：
1. **指数退避**：Per-Message TTL分别设置为1min→2min→4min→8min→16min→30min封顶，每条消息独立设置过期时间（message.setExpiration()），互不影响
2. **DB+MQ双保障**：FallbackFactory既写DB审计（t_notification_retry），又发MQ延迟重试，DB可追溯、MQ高性能
3. **confirmCallback**：生产者开启发送确认回调，消息未送达Broker时记录失败日志
4. **DLQ死信队列**：重试超过5次进入DLQ，人工介入排查

**面试话术**："我们用了RabbitMQ DLX + Per-Message TTL替代了固定间隔轮询。每条消息独立设置过期时间，实现指数退避。第1次1分钟，第2次2分钟，依此类推到30分钟封顶。超过5次进入死信队列，由XXL-Job监控告警。"

#### 3.5 XXL-Job分布式调度

**背景**：@Scheduled注解在单节点运行正常，但多节点部署时重复执行、宕机后任务中断。XXL-Job提供分布式任务调度能力。

**两个任务**：

| 任务名称 | 调度频率 | 职责 |
|---------|---------|------|
| notificationMonitorJob | 每10分钟 | 监控DLQ深度>10条触发告警、统计FAILED记录、计算通知成功率趋势 |
| notificationCleanupJob | 每天2:00 | 清理7天前已SENT的通知记录，避免t_notification_retry表无限增长 |

**技术要点**：
- 执行器注册到调度中心，调度中心负责任务分发
- 支持分片广播（大批量清扫任务可拆分为多个分片并行执行）
- 失败自动重试，避免单次执行失败导致任务中断

#### 3.6 多通道降级策略

**7通道枚举**：SMS（短信）、EMAIL（邮件）、APP_PUSH（App推送）、PHONE（电话）、DINGTALK（钉钉）、WECOM（企业微信）、MANUAL（人工）

**降级链**：
```
SMS失败3次 → 自动降级为PHONE电话通知
  → PHONE也失败 → 终级降级为MANUAL人工介入
```

**实现方式**（NotificationChannel.java）：
```java
public enum NotificationChannel {
    SMS, EMAIL, APP_PUSH, PHONE, DINGTALK, WECOM, MANUAL;

    /** 通道降级链：SMS→PHONE→MANUAL, EMAIL→DINGTALK→MANUAL */
    public NotificationChannel fallbackOf() {
        switch (this) {
            case SMS: return PHONE;
            case EMAIL: return DINGTALK;
            case PHONE: return MANUAL;
            default: return MANUAL;
        }
    }
}
```

**面试话术**："我们设计了7通道降级策略。比如短信发不出去，3次后自动切电话通知；电话也打不通，最终转人工处理。这个设计在金融场景特别重要——告警必须触达，不能因为一个通道故障就丢失。"
