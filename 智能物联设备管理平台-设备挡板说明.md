# 智能物联设备管理平台 — 设备挡板（Mock）业务说明

## 1. 概述

### 1.1 目的

为「智能物联设备管理平台」提供模拟设备实现（Mock Device / 设备挡板），用于以下场景：

- 开发阶段：前端联调、接口自测，无需等待真实硬件到位
- 测试阶段：模拟海量设备并发接入、异常场景（断线、超时、错误数据）
- 演示阶段：可视化设备数据流转，支撑 Demo 演示

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| 协议一致 | 挡板与真实设备使用完全相同的 MQTT Topic 和 HTTP API 格式 |
| 可配置 | 设备型号、数量、上报频率、数据范围均可通过配置文件调整 |
| 行为模拟 | 能模拟正常行为（定时上报）和异常行为（断线重连、数据超阈值、升级失败回滚） |
| 独立部署 | 挡板作为独立进程运行，不侵入平台业务代码 |

---

## 2. 模拟设备类型

根据平台定位的四大行业（金融、能源、工业、消费电子），设计 4 种类型的模拟设备：

| 设备类型 | 产品型号 | 行业 | 典型属性 | 接入协议 |
|----------|----------|------|----------|----------|
| 智能传感器 | SENSOR-TH-001 | 工业 | 温度、湿度、PM2.5、电量 | MQTT |
| 能源采集器 | ENERGY-MTR-001 | 能源 | 电压、电流、功率、累计电量 | MQTT |
| 金融自助终端 | FIN-ATM-001 | 金融 | 运行状态、钞箱余量、网络强度、打印机状态 | HTTP |
| 消费电子网关 | CONSUMER-GW-001 | 消费电子 | CPU使用率、内存使用率、连接设备数、上行流量 | MQTT |

## 3. 挡板功能设计

### 3.1 设备注册挡板

#### 3.1.1 自动注册流程

```
┌──────────┐          ┌──────────┐          ┌──────────┐
│ Mock设备  │          │  MQTT     │          │  平台     │
│          │          │  Broker   │          │          │
└────┬─────┘          └────┬─────┘          └────┬─────┘
     │  1. CONNECT          │                     │
     │─────────────────────►│                     │
     │                      │  2. 校验Token        │
     │                      │────────────────────►│
     │                      │  3. Token有效        │
     │                      │◄────────────────────│
     │  4. CONNACK          │                     │
     │◄─────────────────────│                     │
     │                      │                     │
     │  5. PUBLISH           │                     │
     │  Topic: device/register                      │
     │  Payload: {sn, model, fw_version, ...}      │
     │─────────────────────►│                     │
     │                      │  6. 路由到设备服务    │
     │                      │────────────────────►│
     │                      │  7. 返回 deviceId,   │
     │                      │     deviceSecret     │
     │                      │◄────────────────────│
     │  8. SUBSCRIBE         │                     │
     │  Topic: device/{id}/cmd                     │
     │◄─────────────────────│                     │
```

**挡板行为：**

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1 | 读取配置 | 从 `mock-config.yml` 读取设备类型、SN 前缀、平台地址等参数 |
| 2 | 构造注册报文 | 按设备类型生成符合能力模型的注册 JSON |
| 3 | 建立连接 | MQTT Client 连接 Broker，使用 ClientId = `mock-{deviceType}-{index}` |
| 4 | 发送注册 | 发布到 `device/register` 主题，携带 SN、型号、固件版本 |
| 5 | 保存凭证 | 收到 deviceId 和 deviceSecret 后写入本地缓存，后续通信携带 |
| 6 | 订阅指令 | 订阅 `device/{deviceId}/cmd` 和 `device/{deviceId}/ota` 主题，监听平台指令 |

**注册报文示例（智能传感器）：**

```json
{
  "sn": "SENSOR-TH-20251001-00001",
  "productModel": "SENSOR-TH-001",
  "firmwareVersion": "v1.2.3",
  "hardwareVersion": "H2.0",
  "manufacturer": "RH-Sensor",
  "capabilities": ["temperature", "humidity", "pm25", "battery"],
  "location": {
    "region": "西安",
    "site": "智慧园区-A栋-1层"
  }
}
```

### 3.2 遥测数据上报挡板

#### 3.2.1 定时上报

挡板启动后按配置的频率（默认 30 秒）定时上报遥测数据。

**上报通道：**

| 设备类型 | 上报方式 | Topic / Path |
|----------|----------|--------------|
| 智能传感器 | MQTT Publish | `device/{deviceId}/telemetry` |
| 能源采集器 | MQTT Publish | `device/{deviceId}/telemetry` |
| 金融自助终端 | HTTP POST | `/api/v1/device/{deviceId}/telemetry` |
| 消费电子网关 | MQTT Publish | `device/{deviceId}/telemetry` |

**各设备遥测数据格式：**

**智能传感器：**

```json
{
  "deviceId": "DEV-20251001-00001",
  "timestamp": 1728000000000,
  "data": {
    "temperature": { "value": 26.5, "unit": "℃" },
    "humidity": { "value": 60.2, "unit": "%" },
    "pm25": { "value": 35, "unit": "μg/m³" },
    "battery": { "value": 85, "unit": "%" }
  }
}
```

**能源采集器：**

```json
{
  "deviceId": "DEV-20251001-00101",
  "timestamp": 1728000000000,
  "data": {
    "voltage": { "value": 220.5, "unit": "V" },
    "current": { "value": 15.3, "unit": "A" },
    "power": { "value": 3.37, "unit": "kW" },
    "totalEnergy": { "value": 12580.6, "unit": "kWh" }
  }
}
```

**金融自助终端：**

```json
{
  "deviceId": "DEV-20251001-00201",
  "timestamp": 1728000000000,
  "data": {
    "runningStatus": { "value": "normal" },
    "cashBoxLevel": { "value": 45, "unit": "%" },
    "receiptPaperLevel": { "value": 72, "unit": "%" },
    "networkStrength": { "value": 4, "unit": "bars" },
    "printerStatus": { "value": "ok" }
  }
}
```

**消费电子网关：**

```json
{
  "deviceId": "DEV-20251001-00301",
  "timestamp": 1728000000000,
  "data": {
    "cpuUsage": { "value": 42.3, "unit": "%" },
    "memoryUsage": { "value": 68.7, "unit": "%" },
    "connectedDevices": { "value": 12, "unit": "台" },
    "uplinkTraffic": { "value": 3.2, "unit": "MB/s" }
  }
}
```

#### 3.2.2 数据变异策略

为避免模拟数据过于规律导致测试场景单一，挡板采用以下数据变异策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 正态分布 | 以配置基准值为中心，叠加正态分布随机波动（σ 可配置） | 温度、湿度、电压 等连续量 |
| 缓慢漂移 | 值随时间缓慢递增/递减，模拟设备老化或环境变化 | 电量、累计电量 |
| 随机跳变 | 偶尔产生离群值，模拟传感器故障 | 所有属性 |
| 状态翻转 | 定时切换状态值（normal → warning → normal） | 运行状态、打印机状态 |

**配置示例：**

```yaml
mock-device:
  data-variance:
    temperature:
      base: 26.5
      strategy: normal_distribution
      sigma: 1.5
      min: -10
      max: 60
    battery:
      base: 85
      strategy: slow_drift
      delta: -0.001           # 每次上报降低 0.1%
      min: 0
      max: 100
    runningStatus:
      strategy: state_flip
      states: [normal, warning, error]
      interval: 120            # 每 120 次上报切换一次
      probabilities: [0.85, 0.10, 0.05]
```

### 3.3 设备影子交互挡板

#### 3.3.1 上报属性到影子

挡板每次上报遥测数据后，云端设备影子更新 `reported` 字段。挡板侧无需额外动作，由平台设备影子服务自动完成。

#### 3.3.2 处理期望状态（Desired）

挡板订阅 `device/{deviceId}/cmd` 主题，监听平台下发的期望状态变更。

**处理流程：**

```
平台下发 desired 变更
        │
        ▼
挡板收到 MQTT 消息 (topic: device/{deviceId}/cmd, type: desired_changed)
        │
        ▼
解析 desired JSON，提取变更字段
        │
        ├── temperature: {"target_temperature": 24.0}
        │   └── 调整模拟数据基准值为 24.0，向目标值逐渐逼近
        │
        ├── power_status: "off"
        │   └── 模拟关机流程，发送关机确认，然后断开连接
        │
        └── report_interval: 60
            └── 调整遥测上报频率为 60 秒
        │
        ▼
向平台回执执行结果 (topic: device/{deviceId}/event, type: desired_ack)
```

**desired 变更消息示例：**

```json
{
  "type": "desired_changed",
  "deviceId": "DEV-20251001-00001",
  "version": 16,
  "desired": {
    "target_temperature": 24.0,
    "report_interval": 60
  },
  "timestamp": 1728000100000
}
```

**挡板回执示例：**

```json
{
  "type": "desired_ack",
  "deviceId": "DEV-20251001-00001",
  "version": 16,
  "status": "success",
  "results": [
    { "key": "target_temperature", "status": "accepted", "message": "目标温度设为 24.0℃" },
    { "key": "report_interval", "status": "accepted", "message": "上报间隔调整为 60s" }
  ],
  "timestamp": 1728000110000
}
```

### 3.4 OTA 升级挡板

#### 3.4.1 升级流程模拟

挡板作为 OTA 升级的被控端，模拟完整的固件升级生命周期。平台通过 `device/{deviceId}/ota` 主题下发升级指令。

```
┌──────────┐                              ┌──────────┐
│ Mock设备  │                              │  平台     │
└────┬─────┘                              └────┬─────┘
     │  OTA 指令 (topic: device/{id}/ota)        │
     │◄─────────────────────────────────────────│
     │  Payload: {                               │
     │    firmwareUrl, version, md5, size        │
     │  }                                        │
     │                                           │
     │  1. 上报进度: DOWNLOADING                  │
     │──────────────────────────────────────────►│
     │                                           │
     │  2. 模拟下载 (sleep N秒, N=size/模拟网速)    │
     │  3. 校验 MD5                               │
     │  4. 上报进度: DOWNLOADED                   │
     │──────────────────────────────────────────►│
     │                                           │
     │  5. 模拟安装 (sleep M秒)                    │
     │  6. 上报进度: INSTALLING                   │
     │──────────────────────────────────────────►│
     │                                           │
     │  7. 上报进度: INSTALLED                    │
     │──────────────────────────────────────────►│
     │                                           │
     │  8. 模拟重启                                │
     │  9. 断开 MQTT 连接                          │
     │  10. 等待 N 秒后重连                        │
     │  11. 上报进度: UPGRADE_SUCCESS             │
     │──────────────────────────────────────────►│
```

#### 3.4.2 异常场景模拟

挡板支持通过配置文件注入故障概率，模拟真实设备升级过程中可能出现的异常：

| 异常场景 | 配置项 | 行为 |
|----------|--------|------|
| 下载失败（网络中断） | `download.failRate: 0.05` | 5% 概率上报 DOWNLOAD_FAILED |
| 固件包损坏 | `verify.failRate: 0.02` | 2% 概率 MD5 校验失败，上报 VERIFY_FAILED |
| 安装失败 | `install.failRate: 0.03` | 3% 概率上报 INSTALL_FAILED |
| 重启失败 | `reboot.failRate: 0.01` | 1% 概率重连后仍上报旧版本号 |

```yaml
mock-device:
  ota:
    download:
      speedKbps: 512             # 模拟下载速度
      failRate: 0.05             # 下载失败概率
    verify:
      failRate: 0.02             # 校验失败概率
    install:
      durationSeconds: 30        # 模拟安装耗时
      failRate: 0.03             # 安装失败概率
    reboot:
      durationSeconds: 15        # 模拟重启耗时
      failRate: 0.01             # 重启失败概率
```

**OTA 进度上报报文示例：**

```json
{
  "type": "ota_progress",
  "deviceId": "DEV-20251001-00001",
  "otaTaskId": "OTA-20251001-0001",
  "fromVersion": "v1.2.3",
  "toVersion": "v1.3.0",
  "status": "DOWNLOADING",
  "progress": 45,
  "timestamp": 1728001000000
}
```

**OTA 状态值枚举：**

| 状态 | 说明 |
|------|------|
| DOWNLOADING | 固件下载中，progress 为 0-100 |
| DOWNLOADED | 固件下载完成，MD5 校验通过 |
| DOWNLOAD_FAILED | 固件下载失败 |
| VERIFY_FAILED | 固件包校验失败 |
| INSTALLING | 固件安装中 |
| INSTALLED | 固件安装完成，即将重启 |
| INSTALL_FAILED | 固件安装失败 |
| REBOOTING | 设备重启中 |
| UPGRADE_SUCCESS | 升级成功 |
| UPGRADE_FAILED | 升级失败 |

### 3.5 事件上报与告警挡板

#### 3.5.1 事件上报

挡板按配置的概率和频率向平台上报设备事件，用于触发告警规则。

**上报通道：** MQTT Topic `device/{deviceId}/event` 或 HTTP POST `/api/v1/device/{deviceId}/event`

**智能传感器事件：**

| 事件类型 | eventCode | 触发条件 |
|----------|-----------|----------|
| 高温告警 | TEMP_HIGH | 温度模拟值 > 阈值（如 50℃） |
| 低电量告警 | BATTERY_LOW | 电量模拟值 < 阈值（如 20%） |
| 传感器故障 | SENSOR_FAULT | 按故障概率随机触发 |

**事件报文示例：**

```json
{
  "type": "device_event",
  "deviceId": "DEV-20251001-00001",
  "eventCode": "TEMP_HIGH",
  "eventName": "高温告警",
  "severity": "P1",
  "data": {
    "currentTemperature": 55.3,
    "threshold": 50.0,
    "duration": 120
  },
  "timestamp": 1728001200000
}
```

#### 3.5.2 告警收敛模拟

挡板支持模拟告警风暴场景：在短时间内连续发送同一设备的同一事件，验证平台告警收敛策略是否生效。

```yaml
mock-device:
  event:
    alert-storm:
      enabled: false             # 开启告警风暴模拟
      eventCode: "TEMP_HIGH"
      count: 100                 # 连续发送 100 条
      intervalMs: 100            # 间隔 100ms
```

### 3.6 心跳与离线模拟

#### 3.6.1 心跳机制

**MQTT 设备：** 保持 MQTT 长连接即可，Broker 层面自带心跳检测（Keep Alive）。挡板配置 Keep Alive 间隔（默认 60 秒），平台通过 Last Will Testament 感知设备离线。

**HTTP 设备：** 定时（默认 30 秒）向 `/api/v1/device/{deviceId}/heartbeat` 发送心跳请求。

```json
{
  "deviceId": "DEV-20251001-00201",
  "timestamp": 1728000000000,
  "freeMemory": 128,
  "uptime": 86400
}
```

#### 3.6.2 离线模拟

挡板支持以下几种离线场景模拟：

| 离线场景 | 配置触发方式 | 行为 |
|----------|-------------|------|
| 计划离线 | `offline.scheduled: true, offline.afterSeconds: 300` | 运行 5 分钟后主动断开连接 |
| 心跳超时 | `heartbeat.stopAfterSeconds: 120` | 运行 2 分钟后停止发送心跳 |
| 批量离线（网络分区） | `offline.group: "region-east"` | 整个区域分组同时离线 |
| 断线重连 | `offline.reconnectAfterSeconds: 30` | 离线 30 秒后自动重连 |

```yaml
mock-device:
  offline:
    scheduled: false
    afterSeconds: 0
  heartbeat:
    intervalSeconds: 30
    stopAfterSeconds: 0         # 0 表示不停止
  reconnect:
    enabled: true
    afterSeconds: 30
    maxRetries: 10
```

## 4. 挡板配置总览

### 4.1 完整配置文件

```yaml
# mock-device-config.yml

mock-device:
  # ========== 平台连接配置 ==========
  platform:
    mqttBroker: "tcp://localhost:1883"
    httpBaseUrl: "http://localhost:8080/api/v1"
    authToken: ""               # 预配置的 Token，为空则走自动注册

  # ========== 模拟设备列表 ==========
  devices:
    # 智能传感器 - 模拟 100 台
    - type: SENSOR-TH-001
      count: 100
      snPrefix: "SENSOR-TH"
      protocol: mqtt
      telemetryInterval: 30     # 上报间隔（秒）

    # 能源采集器 - 模拟 50 台
    - type: ENERGY-MTR-001
      count: 50
      snPrefix: "ENERGY-MTR"
      protocol: mqtt
      telemetryInterval: 60

    # 金融自助终端 - 模拟 20 台
    - type: FIN-ATM-001
      count: 20
      snPrefix: "FIN-ATM"
      protocol: http
      telemetryInterval: 30

    # 消费电子网关 - 模拟 30 台
    - type: CONSUMER-GW-001
      count: 30
      snPrefix: "CONSUMER-GW"
      protocol: mqtt
      telemetryInterval: 30

  # ========== 数据变异策略 ==========
  data-variance:
    SENSOR-TH-001:
      temperature:
        base: 26.5
        strategy: normal_distribution
        sigma: 1.5
        min: -10
        max: 60
      humidity:
        base: 60.0
        strategy: normal_distribution
        sigma: 5.0
        min: 0
        max: 100
      pm25:
        base: 35
        strategy: normal_distribution
        sigma: 10
        min: 0
        max: 500
      battery:
        base: 85
        strategy: slow_drift
        delta: -0.001
        min: 0
        max: 100

    ENERGY-MTR-001:
      voltage:
        base: 220.0
        strategy: normal_distribution
        sigma: 5.0
        min: 180
        max: 250
      current:
        base: 15.0
        strategy: normal_distribution
        sigma: 3.0
        min: 0
        max: 100
      power:
        strategy: calculated              # 由 voltage * current 计算
      totalEnergy:
        strategy: slow_drift
        delta: 0.003
        min: 0

    FIN-ATM-001:
      cashBoxLevel:
        strategy: slow_drift
        delta: -0.0005
        min: 0
        max: 100
      runningStatus:
        strategy: state_flip
        states: [normal, warning, error]
        probabilities: [0.85, 0.12, 0.03]
      networkStrength:
        strategy: random_jump
        range: [1, 5]

    CONSUMER-GW-001:
      cpuUsage:
        base: 40.0
        strategy: normal_distribution
        sigma: 15.0
        min: 0
        max: 100
      memoryUsage:
        base: 65.0
        strategy: slow_drift
        delta: 0.0001
        min: 0
        max: 100

  # ========== OTA 配置 ==========
  ota:
    download:
      speedKbps: 512
      failRate: 0.05
    verify:
      failRate: 0.02
    install:
      durationSeconds: 30
      failRate: 0.03
    reboot:
      durationSeconds: 15
      failRate: 0.01

  # ========== 事件上报配置 ==========
  event:
    tempHighThreshold: 50.0              # 高温告警阈值
    batteryLowThreshold: 20.0            # 低电量告警阈值
    sensorFaultRate: 0.001               # 传感器故障概率（每次上报）
    alert-storm:
      enabled: false
      eventCode: "TEMP_HIGH"
      count: 100
      intervalMs: 100

  # ========== 离线/重连配置 ==========
  offline:
    scheduled: false
    afterSeconds: 0
  heartbeat:
    intervalSeconds: 30
    stopAfterSeconds: 0
  reconnect:
    enabled: true
    afterSeconds: 30
    maxRetries: 10

  # ========== 日志配置 ==========
  logging:
    level: INFO
    reportProgress: true                 # 是否打印上报进度
```

### 4.2 启动命令

```bash
# 启动全部 200 台模拟设备
java -jar mock-device-simulator.jar --config=mock-device-config.yml
```

### 5. 内置设备模拟器（新增）

#### 5.1 概述

除外部mock-device模块外，平台Backend内置了 **DeviceSimulator**（`simulator/DeviceSimulator.java`），启动后自动运行，无需额外进程。

#### 5.2 模拟行为

| 阶段 | 行为 | 频率 | 数据链路 |
|------|------|------|---------|
| 启动初始化 | 200台MOCK设备批量上线 + 逐台发送注册消息 | 启动时1次 | register → DeviceService.autoRegister() |
| 遥测上报 | 随机~10台设备上报遥测（温湿度/电压/CPU等），5%概率超阈值 | 每8秒 | telemetry → 写DB+更新影子+发RabbitMQ |
| 告警事件 | 随机~2台设备上报事件（超温/过压/钞箱空/CPU高） | 每8秒 | event → 写DB+RabbitMQ+AlertService匹配规则 |
| 心跳保活 | 批量30台设备心跳 | 每30秒 | heartbeat → DeviceService.handleHeartbeat() |
| 模拟离线 | 随机1-2台设备发送离线事件 | 每90秒 | offline event → 心跳超时→平台检测离线 |

#### 5.3 设计要点

- **复用真实链路**：所有数据经过 `MqttGateway.handleMessage()` → `MqttMessageHandler.dispatch()`，走与真实设备完全相同的业务代码路径
- **符合能力模型**：每种设备类型生成的数据与该产品的能力模型（属性/事件）一致
- **告警验证**：5%概率的超阈值数据确保告警中心持续有新告警流入，便于验证规则匹配和通知补发

### 6. 通知服务挡板（新增）

#### 6.1 目的

为平台的通知功能提供模拟实现，用于验证告警→通知→补发的完整链路。

#### 6.2 挡板信息

| 项目 | 说明 |
|------|------|
| 模块路径 | mock-stubs/ |
| 启动端口 | 8082 |
| 服务名 | iot-notification-service |
| 协议 | HTTP REST |
| Nacos注册 | 支持（enabled=false时可降级） |

#### 6.3 接口说明

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/notification/send | 发送通知（60%成功率模拟真实场景） |
| GET | /api/notification/health | 健康检查 |

#### 6.4 请求/响应示例

请求：
```json
POST /api/notification/send
{
  "alertId": 1,
  "deviceId": "MOCK-SENSOR-0001",
  "channel": "SMS",
  "content": "[IoT告警] 设备MOCK-SENSOR-0001 触发P1级别告警: 温度过高",
  "recipients": "13800138000"
}
```

成功响应(60%概率)：
```json
{"code":200, "message":"success", "data":{"sendId":"SEND-xxx", "status":"SENT", "channel":"SMS"}}
```

失败响应(40%概率)：
```json
{"code":500, "message":"发送失败", "data":{"status":"FAILED", "reason":"运营商返回：号码不在服务区"}}
```

#### 6.5 商业级协作说明：Sentinel熔断 + MQ延迟重试 + XXL-Job监控

通知挡板40%失败率会触发Sentinel熔断规则（1分钟内慢调用>50%→熔断30s）。熔断后Backend的NotificationClientFallback执行DB审计+MQ延迟重试双重保障，不再仅靠定时DB轮询：

**商业级补发调用链：**

```
告警触发 → AlertService.processEvent()
  → @Async sendNotification()
    → NotificationClient.send(url=http://localhost:8082)
      ├─ ✅成功（60%概率）→ Stub返回sendId
      └─ ❌失败（40%概率）/ Sentinel熔断
           → NotificationClientFallback
             ├─ DB: INSERT t_notification_retry (审计)
             └─ MQ: sendWithDelay() → wait.queue(TTL)
                  → TTL过期 → DLX回流 → retry.queue
                    → NotificationRetryConsumer
                      ├─ 成功 → ACK + DB UPDATE SENT
                      └─ 失败 → retryCount++
                         ├─ <5次 → 重新入wait队列 TTL加倍
                         ├─ ≥3次 → 通道降级 SMS→PHONE
                         └─ ≥5次 → 进DLQ人工介入
       → XXL-Job notificationMonitorJob(每10分钟) → 监控DLQ深度>10条告警
       → XXL-Job notificationCleanupJob(每天2点) → 清理7天前SENT
```

**关键设计要点：**

| 机制 | 说明 |
|------|------|
| 指数退避 | Per-Message TTL分别设为1min→2min→4min→8min→16min→30min封顶 |
| 通道降级 | SMS失败3次→自动切PHONE→PHONE也失败→MANUAL人工 |
| DLQ死信 | 超过5次重试进入DLQ死信队列，XXL-Job监控DLQ深度>10条告警 |
| DB+MQ双保障 | DB做审计可追溯，MQ做延迟重试高性能 |
| confirmCallback | RabbitMQ发送确认回调，生产级可靠性 |

**验证方法**：
1. 停掉mock-stubs → 连续触发10条告警 → 观察t_notification_retry表出现10条PENDING记录
2. 同时观察RabbitMQ管理界面wait.queue中有10条待消费消息
3. 启动mock-stubs → 等待TTL过期 → DLX自动转发到retry.queue → NotificationRetryConsumer消费
4. 通知成功→状态变更为SENT；通知再次失败→观察retry_count递增和TTL翻倍
5. 连续失败5次→消息进入DLQ→XXL-Job监控到DLQ深度>0→触发告警
