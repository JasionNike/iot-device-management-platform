### 2.1 设备接入

#### 代码架构

```
DeviceController (REST API)
    ├── POST /api/device/register → DeviceService.manualRegister()
    └── GET  /api/device/list      → DeviceService.listPage()

MqttMessageHandler (MQTT消息分发)
    └── handleRegister() → DeviceService.autoRegister()

DeviceServiceImpl
    ├── autoRegister()   — 自动注册（首次连接时触发）
    ├── manualRegister() — 手动注册（人工录入）
    ├── handleHeartbeat()— 心跳更新
    └── checkOfflineDevices() — 离线检测
```

#### 核心代码位置

- **自动注册**: `service/impl/DeviceServiceImpl.java` 第30-62行，`autoRegister()`方法
- **手动注册**: `service/impl/DeviceServiceImpl.java` 第65-107行，`manualRegister()`方法
- **MQTT入口**: `mqtt/MqttMessageHandler.java` 第72-84行，`handleRegister()`方法
- **REST入口**: `controller/DeviceController.java` 第29-32行，`register()`方法

#### 业务流程

1. **自动注册流程**：设备上电 → MQTT连接 → 发送`register`类型消息 → `MqttMessageHandler.dispatch()`分发 → `handleRegister()`解析JSON → `DeviceService.autoRegister()`查询SN是否已存在 → 不存在则生成deviceId/deviceSecret → 写入`t_device`表 → 初始化空设备影子 → 返回设备身份信息

2. **手动注册流程**：管理员通过UI填写设备信息 → 调用`POST /api/device/register` → `DeviceService.manualRegister()`校验SN唯一性 → 生成设备凭证 → 写入数据库 → 初始化影子

#### 设计亮点

- **自动注册**支持设备"零配置"接入，首次上电即完成注册
- 自动生成16位设备密钥作为设备身份凭证
- 注册同时自动初始化影子，确保影子文档始终存在

### 2.2 设备影子

#### 代码架构

```
ShadowController (REST API)
    ├── GET  /api/shadow/{deviceId}        → ShadowService.getShadow()
    ├── POST /api/shadow/desired/{deviceId}→ ShadowService.setDesired()
    ├── POST /api/shadow/reported/{deviceId}→ ShadowService.updateReported()
    └── DELETE /api/shadow/{deviceId}      → ShadowService.deleteShadow()

ShadowServiceImpl
    ├── RedisTemplate<String, Object> — Redis缓存层
    ├── RedissonClient — 分布式锁
    ├── DeviceShadowMapper — MySQL持久化
    ├── getShadow()         — 三级读取：Redis → MySQL → 空结构
    ├── updateReported()    — 更新设备上报状态（加锁）
    ├── setDesired()        — 设置期望状态（加锁）
    ├── pullAndClearDesired()— 设备拉取期望并清除
    └── deleteShadow()      — 清除缓存和数据库
```

#### 核心代码位置

- **影子读取**: `service/impl/ShadowServiceImpl.java` 第56-88行，`getShadow()`方法
- **上报状态更新**: `service/impl/ShadowServiceImpl.java` 第91-139行，`updateReported()`方法
- **期望状态设置**: `service/impl/ShadowServiceImpl.java` 第142-183行，`setDesired()`方法
- **分布式锁**: 使用Redisson的`RLock`，锁key = `device:lock:{deviceId}`
- **Redis缓存**: key = `device:shadow:{deviceId}`，TTL=7天

#### 业务流程

**reported更新流程**：
设备上报遥测 → `MqttMessageHandler.handleTelemetry()` → `ShadowService.updateReported()` → 获取Redisson分布式锁 → 读取当前影子(Redis/MySQL) → 更新`state.reported`和`metadata.reported`(时间戳) → 版本号+1 → 序列化JSON写入MySQL → 回写Redis缓存 → 释放锁

**desired设置流程**：
平台下发指令 → `POST /api/shadow/desired/{deviceId}` → `ShadowService.setDesired()` → 加锁 → 读取影子 → 更新`state.desired` → 持久化 → 释放锁 → 设备定时拉取`pullAndClearDesired()`获取期望状态并执行

#### 影子JSON结构

```json
{
  "deviceId": "xxx",
  "version": 15,
  "timestamp": 1728000000000,
  "state": {
    "reported": {"temperature": 26.5, "humidity": 60.2},
    "desired": {"temperature": 25.0}
  },
  "metadata": {
    "reported": {"temperature": {"timestamp": 1728000001000}},
    "desired": {"temperature": {"timestamp": 1728000002000}}
  }
}
```

#### 设计亮点

1. **三级影子读取**：Redis → MySQL → 空结构，逐级降级兜底，性能与可靠性兼顾
2. **Redisson分布式锁**：设备ID粒度锁，解决多线程并发更新影子冲突
3. **乐观锁version**：每次更新version+1，MySQL乐观锁插件防CAS覆盖
4. **reported/desired双状态模型**：设备真实状态(reported)与平台期望状态(desired)分离，端云协同基础
5. **metadata时间戳**：每个属性记录更新时间，便于解决时序乱序问题
6. **Redis 7天TTL+主动刷新**：既避免冷数据堆积，又保证热数据高效访问

### 2.3 告警中心

#### 代码架构

```
AlertController (REST API)
    ├── GET  /api/alert/list   → AlertService.listAlerts()
    └── POST /api/alert/handle → AlertService.updateHandleStatus()

AlertServiceImpl
    ├── processEvent()        — 核心：规则匹配+告警收敛+记录生成+MQ通知
    ├── matchCondition()      — JSON条件表达式引擎
    ├── listAlerts()          — 分页查询（按设备/状态筛选）
    ├── updateHandleStatus()  — 处理告警（确认/忽略/修复）
    └── checkAlertEscalation()— 定时升级（P2→P1）

AlertConsumer (RabbitMQ消费者)
    └── handleDeviceEvent()   — 消费设备事件，异步触发告警匹配
```

**数据表**：
- `t_alert_rule` — 告警规则（条件JSON、级别、收敛窗口）
- `t_alert_record` — 告警记录（设备、规则、级别、处理状态）

#### 核心代码位置

- **事件处理入口**: `service/impl/AlertServiceImpl.java` 第41-100行，`processEvent()`方法
- **规则匹配引擎**: `service/impl/AlertServiceImpl.java` 第163-205行，`matchCondition()`方法
- **告警收敛**: `service/impl/AlertServiceImpl.java` 第58-68行
- **超时升级**: `service/impl/AlertServiceImpl.java` 第138-157行，`checkAlertEscalation()`方法
- **MQ消费者**: `mq/AlertConsumer.java` 第34-58行

#### 业务流程

1. **告警触发流程**：设备上报事件 → `MqttMessageHandler.handleEvent()` → `EventProducer.sendDeviceEvent()`（发MQ） + `AlertService.processEvent()`（同步） → 查询所有启用规则 → 逐个匹配condition_json → 收敛检查(同设备+同规则+30分钟) → 生成告警记录 → 发送告警MQ通知

2. **告警处理流程**：运维人员查看告警列表 → `POST /api/alert/handle` → `updateHandleStatus()`更新处理状态+处理人+备注+处理时间

3. **告警升级流程**：定时任务(`@Scheduled(fixedRate=300000)`)每5分钟执行 → 查询P2+PENDING+创建时间超过30分钟的告警 → 自动升级为P1

#### 条件匹配引擎

告警规则的`condition_json`字段支持灵活的条件配置：
```json
// 温度超过60度告警
{"property": "temperature", "operator": ">", "threshold": 60}
// 状态不等于NORMAL告警
{"property": "status", "operator": "!=", "threshold": "NORMAL"}
```

支持的运算符：`>` `>=` `<` `<=` `==` `!=`，自动识别数值类型和字符串类型比较。

#### 设计亮点

1. **规则引擎轻量化**：通过JSON条件表达式实现灵活告警规则配置，无需硬编码
2. **告警收敛**：30分钟窗口内重复告警自动合并，避免告警风暴
3. **P2→P1自动升级**：超时未处理的告警自动升级，确保关键问题不被遗漏
4. **MQ异步解耦**：告警规则匹配通过RabbitMQ异步执行，不阻塞设备数据上报主流程

### 2.4 OTA升级

#### 代码架构

```
OtaController (REST API)
    ├── GET  /api/ota/firmwares         → firmwareMapper.selectList()  固件列表
    ├── POST /api/ota/firmware          → firmwareMapper.insert()      上传固件
    ├── POST /api/ota/task              → OtaService.createTask()
    ├── POST /api/ota/gray/{taskId}     → OtaService.advanceGrayPercent()
    ├── POST /api/ota/progress          → OtaService.updateProgress()
    ├── GET  /api/ota/records/{deviceId} → OtaService.getDeviceOtaRecords()
    └── GET  /api/ota/tasks             → OtaService.listTasks()

OtaServiceImpl
    ├── createTask()          — 创建OTA任务（灰度比例/目标产品/时间窗口）
    ├── advanceGrayPercent()  — 灰度推进（+10%/次）
    ├── updateProgress()      — 设备上报进度（含失败自动重试）
    ├── getDeviceOtaRecords() — 查询设备OTA历史
    └── listTasks()           — 任务列表
```

**数据表**：
- `t_ota_task` — 升级任务（灰度百分比、固件ID、目标类型/值、时间窗口）
- `t_ota_record` — 升级记录（设备、任务、状态、进度、重试次数）
- `t_firmware` — 固件管理（版本、文件路径、MD5）

#### 核心代码位置

- **创建任务**: `service/impl/OtaServiceImpl.java` 第26-34行，`createTask()`方法
- **灰度推进**: `service/impl/OtaServiceImpl.java` 第38-50行，`advanceGrayPercent()`方法
- **进度上报**: `service/impl/OtaServiceImpl.java` 第53-88行，`updateProgress()`方法
- **OTA记录查询**: `service/impl/OtaServiceImpl.java` 第91-96行

#### 业务流程

1. **创建升级任务**：管理员上传固件 → 创建OTA任务(选择产品、固件版本、灰度比例) → 写入`t_ota_task`

2. **灰度升级**：初次创建灰度比例低(如10%) → 运维确认稳定后调用`advanceGrayPercent()` → 每次增加10% → 到达100%自动标记COMPLETED

3. **设备端上报进度**：设备收到升级指令 → 下载固件 → 安装 → 重启 → 每步通过`updateProgress()`上报 → 服务端记录状态 → 失败且retryCount<3自动置为PENDING等待重试

#### 设计亮点

1. **灰度发布**：支持按百分比灰度，可逐步推进降低升级风险
2. **失败自动重试**：升级失败自动重试最多3次，提高升级成功率
3. **完整升级记录**：每次升级操作都有完整轨迹可追溯

### 2.6 Redis缓存与分布式锁方案

#### 体系架构

```
┌─────────────────────────────────────────────────┐
│                  ShadowServiceImpl              │
│  ┌─────────────┐   ┌──────────┐   ┌──────────┐ │
│  │ Redis缓存   │   │ Redisson │   │  MySQL   │ │
│  │ 7天TTL+刷新 │   │ 分布式锁 │   │ 持久化   │ │
│  └─────────────┘   └──────────┘   └──────────┘ │
│        │                │               │       │
│  device:shadow:   device:lock:      t_device_   │
│  {deviceId}       {deviceId}       _shadow      │
└─────────────────────────────────────────────────┘
```

#### 配置位置

- **Redis连接**: `resources/application.yml` 第18-24行
- **Redis序列化**: `config/RedisConfig.java` 第21-47行（Jackson2JsonRedisSerializer）
- **分布式锁**: `service/impl/ShadowServiceImpl.java` 使用Redisson

#### 设计亮点

1. **双重缓存策略**：Redis作为一级缓存(7天TTL+主动刷新)，MySQL作为持久化存储
2. **设备级分布式锁**：Redisson锁key粒度到deviceId，避免全局锁性能瓶颈
3. **乐观锁双重保障**：version号CAS更新+MyBatis-Plus乐观锁插件
4. **Jackson2JsonRedisSerializer**：自动序列化/反序列化影子JSON，保留泛型类型信息

  └── @Async → CompletableFuture.runAsync() → Feign调用通知服务

调用链路（商业级完整闭环）：
```
告警触发 → AlertService.processEvent()
  → @Async sendNotification()
    → NotificationClient.send(url=http://localhost:8082)
      ├─ ✅成功 → Notification Stub返回sendId
      └─ ❌失败/Sentinel熔断
           → NotificationClientFallback
             ├─ DB: INSERT t_notification_retry (审计)
             └─ MQ: sendWithDelay() → wait.queue(TTL)
                  → TTL过期 → DLX回流 → retry.queue
                    → NotificationRetryConsumer
                      ├─ 成功 → ACK + DB UPDATE SENT
                      └─ 失败 → retryCount++

                         └─ ≥5次 → 进DLQ人工介入
       → XXL-Job notificationMonitorJob(每10分钟) → 监控DLQ/FAILED/成功率
       → XXL-Job notificationCleanupJob(每天2点) → 清理7天前SENT
```
