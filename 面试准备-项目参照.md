# 智能物联设备管理平台 — 面试项目参照

## 一、项目概述

面向金融、能源、工业与消费电子等行业的智能终端统一接入与运营平台。依托润和在OpenHarmony/HarmonyOS Connect领域的生态能力，支撑设备注册、能力模型管理、OTA升级、运行监控、告警联动和端云协同等业务场景。核心解决三大痛点：多厂商设备协议不统一、海量设备状态同步延迟、运维闭环难。

项目时间：2025.09 - 2026.02（6个月）

## 二、业务范围

1. **设备接入**：支持MQTT/HTTP双协议，设备自动注册/手动注册/批量导入，设备认证（Token签名）
2. **设备影子**：云端维护设备状态JSON文档（reported+desired双状态），解决端云同步和离线指令问题
3. **能力模型管理**：产品型号定义，属性(Property)/事件(Event)/指令(Command)管理
4. **OTA升级**：固件包管理、灰度发布（5%→20%→50%→100%分批）、失败重试+回滚
5. **告警中心**：阈值/状态/事件告警，P0-P3四级，RabbitMQ消费设备事件，告警闭环处理，收敛防风暴
6. **运行监控**：设备列表/详情/统计大盘、心跳检测（3次超时判离线）

## 三、我负责参与的功能

- **设备接入模块**：MQTT/HTTP协议适配，设备注册与认证，密钥分发

- **告警中心**：告警规则引擎，基于RabbitMQ异步消费设备事件→匹配规则→生成告警，告警收敛防风暴

- **设备模拟器**：内置DeviceSimulator，启动时200台MOCK设备注册→每8秒遥测+事件+心跳→全链路真实数据流

## 四、核心业务流程

### 设备影子同步流程

```
设备上报遥测数据(MQTT)
  → MqttGateway接收 → MqttMessageHandler.dispatch()
  → 写t_device_telemetry遥测表
  → 更新设备影子ShadowService.updateReported()
    → 获取Redisson分布式锁(设备ID粒度)
    → 读当前影子JSON(Redis优先，未命中查MySQL)
    → 更新reported字段 + metadata时间戳
    → 版本号version+1（乐观锁）
    → 写Redis缓存(7天TTL)
    → 持久化MySQL
    → 释放锁

平台下发指令
  → ShadowService.setDesired()
    → 更新desired字段
    → 设备下次上线拉取差异并执行
```

### 告警触发流程

```
设备上报事件(MQTT) → MqttMessageHandler处理
  → 写t_device_event表
  → RabbitMQ发送设备事件
  → AlertConsumer异步消费
  → 查t_alert_rule启用规则
  → matchCondition()匹配（支持>/</==/>=/<=运算符）
  → 告警收敛检查（同设备同规则30分钟内去重）
  → 生成告警记录（P0-P3等级）
  → 写入t_alert_record
  → 多渠道通知（短信/邮件/钉钉）
```

## 六、技术选型理由

### 为什么用MQTT而不是HTTP？

MQTT是物联网事实标准协议，基于TCP长连接，报文头部仅2字节，支持QoS 0/1/2三级服务质量。相比HTTP轮询：省带宽（二进制报文）、实时性好（长连接推送）、支持遗嘱消息（设备异常断开自动通知）。但HTTP也有其适用场景——不支持长连接的设备用HTTP定时上报。

### 为什么设备影子用Redis String而不是Hash？

影子是完整JSON文档，读写都是整取整存，String类型最合适。如果用Hash，需要把JSON拍平为多个field，复杂且不利于version管理。Redis String + JSON序列化/反序列化（Jackson），简单高效。

### 为什么告警规则用matchCondition自己实现而不是用规则引擎（如Drools）？

6条规则用Drools太重了。自定义matchCondition方法解析condition_json（property+operator+threshold），支持简单比较运算，代码不到30行。如果规则上百条且逻辑复杂，才考虑引入规则引擎。

## 八、面试官可能问到的问题

### Q1: 设备影子为什么用分布式锁？什么场景下会并发？

同一设备可能同时收到：①设备上报状态 ②运维人员下发指令。两个请求并发更新影子JSON时会产生覆盖。用Redisson分布式锁保证同一设备串行写入，锁粒度是deviceId级别，不同设备互不阻塞。

### Q2: 告警收敛怎么做的？

同一设备+同一规则在10分钟窗口内不重复创建告警。实现：创建告警前查t_alert_record（device_id=xxx AND rule_id=xxx AND create_time > now-10min），有记录则跳过。避免一个高温设备每分钟发一次告警的"告警风暴"。

### Q3: OTA灰度发布怎么实现？

4阶段分批：5%→20%→50%→100%。每阶段观察2-12小时，检查设备崩溃率/运行状态，指标达标后自动推进下一阶段。灰度命中用设备ID哈希取模（device_id.hashCode() % 100 < percentage），稳定且分布均匀。

### Q4: 设备和云端的时间不同步怎么办？

每次设备连接后执行NTP时间同步，以服务器时间为准。设备上报数据时携带设备时间戳，云端记录metadata中的timestamp。离线设备下次上线时先同步NTP，再处理积压数据。

### Q5: 设备离线后下发的指令怎么处理？

指令存到影子的desired字段。设备上线后调pullAndClearDesired()，拉取desired与本地状态做diff，执行变更，上报新的reported，同时清除已执行的desired。用version号标记每次变更，防止重复执行。

### Q6: 为什么用Redisson而不是直接用Redis SET NX？

Redisson封装了：Watchdog自动续期（每10秒）、可重入锁（Redis Hash实现）、公平锁、读写锁、信号量。自己用SET NX实现需要处理续期、重入、释放时校验持有者等边界条件，代码量大且容易出Bug。Redisson生产验证充分，社区活跃。

### Q7: 设备量大了怎么扩展？

1) MQTT Broker集群（EMQX支持水平扩展）；2) Redis集群分片；3) 影子数据按deviceId哈希分库分表；4) RabbitMQ消费端水平扩展；5) 告警规则匹配可以异步批量处理。

### Q8: 为什么用Gateway做统一入口而不是直接暴露Backend？

①前端只需知道一个端口(:8080)，内部拓扑对外透明；②认证/限流/日志在网关层统一处理，业务服务不需重复实现；③未来服务拆分时前端零改动，只需修改网关路由规则；④TraceId在网关层生成并透传，全链路追踪。

### Q9: Sentinel的熔断降级和Fallback怎么配合使用？

@SentinelResource注解配置两个兜底方法——fallback处理业务异常(如数据库挂了返回缓存数据)，blockHandler处理限流降级(如QPS超限返回"稍后再试")。Feign调用用fallbackFactory获取异常原因，将失败通知持久化到t_notification_retry表等补发。

### Q10: 通知服务不可用时，告警通知怎么保证不丢失？（商业级MQ+DB双保障）

采用五层保障机制确保告警通知不丢失，从Demo级的"DB轮询"升级为商业级的"MQ延迟重试+XXL-Job监控"：

1. **Feign直连降级**：NotificationClient配置`url="${iot.notification.service-url:http://localhost:8082}"`，Nacos不可用时自动降级为直连Notification Stub，避免服务发现故障导致通知断裂

2. **Sentinel熔断+FallbackFactory（DB+MQ双保障）**：通知服务异常率>50%时熔断30s（快速失败避免线程阻塞），FallbackFactory不再仅是写DB，而是DB审计+MQ延迟双重保障：
   - **DB审计**：INSERT t_notification_retry记录失败原因和上下文，用于追溯
   - **MQ延迟重试**：sendWithDelay()将消息发到wait.queue，通过Per-Message TTL实现指数退避

3. **RabbitMQ DLX延迟重试（核心升级）**：替代原来的@Scheduled固定2分钟DB轮询：
   - **架构**：DirectExchange → waitQueue（设TTL）→ TTL过期 → DLX回流 → retryQueue → 消费者处理
   - **指数退避**：1min→2min→4min→8min→16min→30min封顶，每次重试TTL翻倍
   - **Per-Message TTL**：每条消息独立设置过期时间，而非队列统一TTL，保证每条消息退避节奏独立
   - **confirmCallback**：生产者开启发送确认回调，消息送达Broker才视为成功

4. **通道降级策略**：NotificationRetryConsumer消费retry.queue时，根据retryCount执行通道降级：
   - retryCount<3次：继续原通道重试
   - retryCount≥3次：自动降级（SMS→PHONE电话通知）
   - PHONE也失败→MANUAL人工介入
   - 7通道枚举：SMS/EMAIL/APP_PUSH/PHONE/DINGTALK/WECOM/MANUAL

5. **XXL-Job分布式监控**：替代@Scheduled，解决单机定时任务宕机失效问题：
   - notificationMonitorJob（每10分钟）：监控DLQ深度>10条触发告警、统计FAILED记录、计算成功率
   - notificationCleanupJob（每天2点）：清理7天前SENT记录

完整闭环链路：

```
告警触发 → AlertService.processEvent()
  → @Async sendNotification()
    → NotificationClient.send(url=http://localhost:8082)
      ├─ ✅成功 → Notification Stub返回sendId
      └─ ❌失败/Sentinel熔断
           → NotificationClientFallback
             ├─ DB: INSERT t_notification_retry (审计)
             └─ MQ: sendWithDelay() → wait.queue(TTL)
                  → TTL过期 → DLX回流 → retry.queue
                    → NotificationRetryConsumer
                      ├─ 成功 → ACK + DB UPDATE SENT
                      └─ 失败 → retryCount++
                         ├─ <5次 → 重新入wait队列 TTL加倍
                         ├─ ≥3次 → 通道降级 SMS→PHONE
                         └─ ≥5次 → 进DLQ人工介入
       → XXL-Job notificationMonitorJob(每10分钟) → 监控DLQ/FAILED/成功率
       → XXL-Job notificationCleanupJob(每天2点) → 清理7天前SENT
```

### Q11: 开发/测试阶段没有真实设备，数据怎么来？

**回答**：我们内置了DeviceSimulator设备模拟器，启动后自动模拟200台MOCK设备。模拟器走的是与真实设备完全相同的代码路径——`MqttGateway.handleMessage()` → `MqttMessageHandler.dispatch()`。每8秒随机选取~15台设备上报遥测（其中5%概率生成超阈值数据触发告警），每30秒批量心跳，每90秒模拟设备离线。这样仪表盘和告警中心始终有实时变化的数据，开发联调和演示都不需要真实硬件。

### Q12: 通知重试的指数退避怎么实现的？为什么用Per-Message TTL？

采用RabbitMQ DLX（死信交换机）+ Per-Message TTL实现非等间隔延迟：

**架构设计**：
```
sendWithDelay(msg, delayMs)
  → rabbitTemplate.convertAndSend("notification.direct", "wait.rk", msg, message -> {
        // 1. 每条消息独立设置过期时间，互不影响
        message.getMessageProperties().setExpiration(String.valueOf(delayMs));
        // 2. 设置死信路由，TTL过期后自动转发到retry.queue
        message.getMessageProperties().setHeader("x-retry-count", retryCount);
        return message;
    });
  → 消息进入 wait.queue（无消费者，仅做延时缓冲）
  → TTL到期 → 自动转发到 notification.dlx（死信交换机）
  → DLX按routing key"retry.rk"路由 → 进入 retry.queue
  → NotificationRetryConsumer 消费处理
```

**指数退避算法**（`NotificationMessage.getBackoffDelayMs()`）：
```
// 第1次重试: 1分钟 → 第2次: 2分钟 → 第3次: 4分钟
// → 第4次: 8分钟 → 第5次: 16分钟 → 封顶30分钟
long delay = Math.min((long) (1000 * 60 * Math.pow(2, retryCount)), 30 * 60 * 1000);
```

**为什么不用队列统一TTL？**
- 统一TTL（`x-message-ttl`）所有消息相同过期时间，无法实现指数级递增
- Per-Message TTL每条消息独立设置expiration，第1次1分钟、第2次2分钟、第3次4分钟……完全符合指数退避需求

**为什么不用@Scheduled+DB轮询？**
- 商业级可靠性要求：@Scheduled单机执行，宕机后补发中断
- 精准时间控制：DB轮询最小粒度1分钟，Per-Message TTL精确到毫秒
- 性能优势：MQ推送模式，无需频繁扫表浪费数据库IO

**confirmCallback保障**：
```java
rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
    if (!ack) {
        log.error("消息发送到Broker失败, cause={}", cause);
        // 记录到DB，人工介入
    }
});
```

## 九、Java高级开发技术面试高频问题

### 🔴 MQTT vs HTTP vs WebSocket（结合项目）

**Q: 为什么选MQTT？MQTT的QoS等级怎么理解？**

**回答**：MQTT是IoT标准协议，基于TCP长连接，报文头部仅2字节。QoS 0（至多一次）适合不重要数据；QoS 1（至少一次）保证到达但可能重复；QoS 2（恰好一次）四次握手保证不丢不重但开销最大。对比HTTP：MQTT长连接实时推送、二进制省带宽、支持遗嘱消息。对比WebSocket：MQTT的QoS和Topic通配符是WebSocket不具备的。我们大部分设备用QoS 1，关键指令用QoS 2。

### 🔴 Netty 与 MQTT Broker（高频）

**Q: MQTT协议底层怎么实现高性能异步通信？**

**回答**：MQTT Broker底层通常基于Netty的Reactor模型。主从Reactor线程池：BossGroup处理连接，WorkerGroup处理读写。I/O多路复用（epoll）使得单线程能管理成千上万个连接。Netty的零拷贝（DirectBuffer、CompositeByteBuf、FileRegion）减少内存复制。我们项目中用spring-integration-mqtt封装了底层实现。

### 🔴 百万设备同时在线的架构设计（系统设计）

**Q: 100万设备同时在线，怎么设计？**

**回答**：①MQTT Broker集群（EMQX支持水平扩展，每节点20万连接=5个节点）；②设备影子用Redis集群分片（按deviceId哈希）；③设备状态数据用时序数据库（InfluxDB/TDengine）替代MySQL；④告警规则匹配用流处理（Flink CEP）；⑤RabbitMQ消费端水平扩展，prefetch限流；⑥Nginx/HAProxy做MQTT接入层负载均衡。

### 🔴 分布式锁与并发控制（结合项目）

**Q: 设备影子的并发写入除了分布式锁还有什么方案？**

**回答**：除了Redisson分布式锁，还有：①乐观锁version号（CAS校验，更新WHERE version=旧值）；②AtomicReference（JVM内的原子引用，仅适用于单节点）；③ZooKeeper（CP强一致但太重）。我们的方案是双层保护：Redisson分布式锁（第一层）+ 影子version乐观锁（第二层），即使分布式锁短暂失效，version校验也能拦截并发覆盖。

### 🔴 OTA灰度发布的哈希算法（结合项目）

**Q: 灰度百分比怎么保证用户分布的均匀性和稳定性？**

**回答**：用deviceId的hashCode取模：`Math.abs(deviceId.hashCode()) % 100 < grayPercent`。Java String的hashCode算法是`s[0]*31^(n-1) + s[1]*31^(n-2) + ...`，分布足够均匀。稳定性：同一deviceId的hashCode固定，所以同一设备每次计算结果一致，不会出现"上次命中灰度、这次不命中"的问题。

### 🔴 告警风暴与消息堆积（结合项目）

**Q: RabbitMQ消息堆积了怎么办？**

**回答**：告警风暴场景（如100台设备同时超温）：①告警收敛——同设备同规则10分钟内只生成一条告警；②消费者prefetch限流——每次最多处理50条；③消费者批量处理——每次100条批量写DB；④队列TTL+死信队列兜底；⑤监控队列深度，超过阈值自动扩容消费者。IoT项目的告警收敛通过Redis记录(deviceId+ruleId+时间窗口)实现。

### 🔴 Redis数据结构底层（高频）

**Q: ZSet的底层实现？为什么用跳表不用红黑树？**

**回答**：ZSet在数据少时用ziplist（压缩列表），数据多时用skiplist（跳表）+dict（字典）。跳表比红黑树的优势：①实现简单，容易维护；②范围查询O(logN+M)可以直接遍历链表；③红黑树需要中序遍历，实现复杂。字典用于O(1)的按member查score，跳表用于O(logN)的按score排序。

### 🔴 定时任务的分布式互斥（结合项目）

**Q: XXL-Job怎么保证定时任务在多节点不重复执行？**

**回答**：XXL-Job通过路由策略保证：①"第一个"策略——调度中心只给第一个注册的执行器分配任务；②"一致性Hash"——相同任务路由到相同执行器。原理：执行器启动时向调度中心注册，调度中心维护执行器列表，任务触发时按路由策略选择执行器。不需要额外引入分布式锁。
