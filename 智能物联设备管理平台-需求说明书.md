# 智能物联设备管理平台 — 需求说明书

## 1. 文档概述

| 项目 | 说明 |
|------|------|
| 项目名称 | 智能物联设备管理平台 |
| 版本 | V1.0 |
| 目标行业 | 金融、能源、工业、消费电子 |
| 技术生态 | OpenHarmony / HarmonyOS Connect |

### 1.1 项目背景

面向金融自助终端、能源采集器、工业传感器、消费电子设备等多厂商异构智能终端场景，建设统一的设备接入与运营管理平台。依托润和在 OpenHarmony / HarmonyOS Connect 领域的生态能力，重点解决以下痛点：

- 多厂商设备协议不统一（MQTT / HTTP / 私有协议）
- 设备量大，海量终端同时在线管理与实时状态同步
- 状态同步延迟高，端云数据不一致
- 运维闭环困难，告警 → 定位 → 修复链路断裂

## 2. 功能需求

### 2.1 设备接入模块

#### 2.1.1 设备注册

| 编号 | FR-DEV-001 |
|------|-------------|
| 名称 | 设备注册 |
| 优先级 | P0 |
| 描述 | 支持设备首次上线时自动注册，或通过管理后台手动录入设备信息。注册后生成全局唯一设备标识（deviceId），并初始化设备元数据。 |

**功能点：**

1. **自动注册**：设备上电后通过 MQTT / HTTP 上报设备序列号（SN）、型号、固件版本等信息，平台自动创建设备记录并分配 deviceId。
2. **手动注册**：运营人员在管理后台填写设备名称、所属产品型号、部署位置、设备 SN 等信息完成注册。
3. **设备认证**：注册时下发 deviceSecret，后续设备通信需携带 Token（deviceId + deviceSecret + timestamp 签名），平台校验通过后方可交互。
4. **设备分组**：支持按产品型号、固件版本、部署区域、在线状态等维度分组管理。
5. **批量导入**：支持 Excel/CSV 模板批量导入设备信息，适用于设备出厂前预注册。

**数据实体：**

| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | varchar(64) | 全局唯一设备 ID |
| device_name | varchar(128) | 设备名称 |
| product_id | varchar(64) | 所属产品型号 ID |
| device_sn | varchar(128) | 设备出厂序列号 |
| device_secret | varchar(256) | 设备认证密钥 |
| firmware_version | varchar(32) | 当前固件版本 |
| location | varchar(256) | 部署位置 |
| online_status | tinyint | 在线状态 0-离线 1-在线 |
| register_time | datetime | 注册时间 |
| last_online_time | datetime | 最近上线时间 |

#### 2.1.2 设备接入协议适配

| 编号 | FR-DEV-002 |
|------|-------------|
| 名称 | 多协议接入适配 |
| 优先级 | P0 |
| 描述 | 平台需同时支持 MQTT 和 HTTP 两种主流设备通信协议，具备协议适配层，对外暴露统一接口。 |

**功能点：**

1. **MQTT 接入**：设备通过 MQTT Broker 建立长连接，上报遥测数据和事件，接收平台下发的指令和配置。
2. **HTTP 接入**：适用于不支持长连接的设备，通过 HTTP POST 定时上报数据，通过 HTTP GET 轮询拉取平台下发的待执行指令。
3. **协议转换层**：在网关层实现协议解析与转换，将 MQTT Topic / HTTP Path 映射为统一内部消息格式，上层业务模块无感知协议差异。

### 2.2 设备影子模块

| 编号 | FR-SHADOW-001 |
|------|---------------|
| 名称 | 设备影子 |
| 优先级 | P0 |
| 描述 | 为每个设备在云端维护一份 JSON 格式的状态文档（影子），缓存设备的最新上报状态及平台侧期望状态，解决端云状态同步延迟和离线场景下的指令下发问题。 |

**功能点：**

1. **状态同步**：设备上报属性数据（如温度、电量、开关状态等），实时更新影子中的 `reported` 字段。
2. **期望状态**：平台向设备下发期望状态（如设置目标温度、远程开关机），写入影子的 `desired` 字段。设备上线后自动拉取差异并执行。
3. **版本管理**：影子每次更新递增 version 号，支持乐观锁防止并发覆盖。
4. **增量同步**：设备上报时仅携带变更字段，平台计算 diff 后仅更新变化部分，减少网络开销。
5. **离线指令暂存**：设备离线时平台下发的期望状态暂存于影子中，设备下次上线时自动拉取执行，执行完成后清除 desired。

**影子 JSON 结构示例：**

```json
{
  "deviceId": "DEV-20251001-00001",
  "version": 15,
  "timestamp": 1728000000000,
  "state": {
    "reported": {
      "temperature": 26.5,
      "humidity": 60.2,
      "power_status": "on",
      "battery": 85
    },
    "desired": {
      "target_temperature": 24.0,
      "power_status": "off"
    }
  },
  "metadata": {
    "reported": {
      "temperature": { "timestamp": 1728000000000 },
      "humidity": { "timestamp": 1728000000000 }
    },
    "desired": {
      "target_temperature": { "timestamp": 1727999900000 }
    }
  }
}
```

**缓存策略：**

- 设备影子数据存入 Redis（String 类型，key = `device:shadow:{deviceId}`），设置 7 天过期 + 主动刷新策略。
- 高频查询直接从 Redis 读取，避免直接访问数据库。
- 设备离线超过 7 天未上报，影子自动过期，下次上线重新初始化。

### 2.3 能力模型管理模块

| 编号 | FR-CAP-001 |
|------|-------------|
| 名称 | 能力模型管理 |
| 优先级 | P1 |
| 描述 | 定义每种产品型号的设备能力模型，包括属性（Property）、事件（Event）、指令（Command）三类能力，统一定义设备数据交互契约。 |

**功能点：**

1. **产品型号管理**：创建产品型号（Product），定义产品名称、厂商、设备类型（传感器 / 网关 / 控制器 / 消费终端）。
2. **属性定义**：为产品定义属性列表（如 温度、湿度、开关状态），包括属性名、数据类型、单位、读写权限、取值范围。
3. **事件定义**：定义设备可上报的事件类型（如 高温告警、低电量告警、设备离线），包括事件参数和输出参数。
4. **指令定义**：定义平台可向设备下发的指令（如 重启、设置阈值、远程锁屏），包括输入参数和是否异步执行。
5. **模型版本管理**：能力模型支持版本号，设备注册时绑定产品型号+模型版本，升级固件时可同步更新能力模型。

**数据实体：**

```
Product（产品型号）
 ├── Property（属性）       1:N
 ├── Event（事件）          1:N
 └── Command（指令）        1:N
```

### 2.4 OTA 升级模块

| 编号 | FR-OTA-001 |
|------|-------------|
| 名称 | OTA 固件升级 |
| 优先级 | P0 |
| 描述 | 支持对设备进行远程固件升级（Over-The-Air），包括固件包管理、升级任务创建、灰度策略、升级进度监控和失败回滚。 |

**功能点：**

1. **固件包管理**：上传固件包（.bin / .zip），关联产品型号和目标版本号，记录 MD5 校验值、文件大小和上传时间。
2. **升级任务创建**：选择目标固件版本，指定升级范围（全量 / 按分组 / 按设备列表），设置升级时间窗口。
3. **灰度发布**：支持按百分比（如 10% → 50% → 100%）分批升级，每批升级完成后观察设备运行状态，确认无误后再推下一批。
4. **升级状态跟踪**：设备端分阶段上报升级进度（下载中 → 下载完成 → 安装中 → 安装完成 → 重启 → 升级成功/失败），平台记录。
5. **失败重试与回滚**：升级失败的设备自动重试最多 3 次，仍失败则标记为升级失败并通知运维人员，支持手动回滚到上一稳定版本。
6. **升级统计**：统计升级成功率、平均升级耗时、失败原因分布。

**OTA 状态机：**

```
待下载 → 下载中 → 下载完成 → 安装中 → 安装完成 → 重启中 → 升级成功
  ↓                  ↓                    ↓
下载失败          安装失败            重启失败 → 自动重试(3次) → 升级失败
```

### 2.5 运行监控模块

| 编号 | FR-MON-001 |
|------|-------------|
| 名称 | 设备运行监控 |
| 优先级 | P0 |
| 描述 | 实时监控设备在线状态、遥测数据、运行指标，提供设备列表视图、设备详情看板和统计大盘。 |

**功能点：**

1. **设备列表**：展示全部设备，支持按产品型号、在线状态、部署区域、固件版本筛选，支持设备名称/SN 模糊搜索，分页加载。
2. **设备详情**：查看单个设备的实时遥测数据、历史数据趋势图、告警记录、OTA 升级记录、操作日志。
3. **统计大盘**：
   - 设备总数、在线/离线设备数、在线率趋势
   - 按产品型号的设备分布
   - 近 24 小时告警数量及等级分布
   - OTA 升级进度统计
4. **心跳检测**：设备定期发送心跳（MQTT PINGREQ / HTTP heartbeat），超时 3 次未收到心跳则判定离线，触发离线告警。
5. **数据存储**：遥测历史数据存入 MySQL（按天分表），保留 90 天；超期数据归档至冷存储。

### 2.6 告警中心模块

| 编号 | FR-ALERT-001 |
|------|---------------|
| 名称 | 告警中心 |
| 优先级 | P0 |
| 描述 | 基于设备上报事件和平台监控规则，生成告警记录，支持告警分级、告警通知、告警确认/处理/关闭的完整闭环。 |

**功能点：**

1. **告警规则配置**：支持配置阈值告警（温度 > 60℃）、状态告警（设备离线超过 5 分钟）、事件告警（设备上报故障码），支持与、或条件组合。
2. **告警等级**：P0-紧急、P1-严重、P2-一般、P3-提示。不同等级差异化通知策略。
3. **告警生成**：规则引擎实时消费设备事件消息（RabbitMQ），匹配告警规则，生成告警记录写入数据库，同时更新 Redis 告警计数缓存。
4. **告警通知**：P0/P1 告警通过短信、邮件、企业微信等方式通知相关运维人员；P2/P3 告警在平台内展示。
5. **告警处理闭环**：
   - 待处理 → 已确认（运维人员接单）→ 处理中 → 已解决（填写处理备注）→ 已关闭
   - 支持告警升级：P2 告警超时未处理自动升级为 P1
6. **告警收敛**：同一设备同一规则的重复告警，在时间窗口内（默认 10 分钟）进行收敛合并，避免告警风暴。
7. **告警统计**：告警数量趋势、处理时效统计、告警类型分布。

| 描述 | 平台采用Spring Cloud Gateway作为统一API入口(:8080)，所有外部请求经过网关路由转发至后端IoT核心服务(:8081)和通知服务(:8082)。 |

### 2.8 熔断降级模块（新增）

| 编号 | FR-SENTINEL-001 |
| 名称 | Sentinel熔断降级与限流 |
| 优先级 | P0 |
| 描述 | 引入Alibaba Sentinel实现四维防护：接口级流控、Feign熔断降级、热点参数限流、系统自适应保护。 |

功能点：

4. 系统规则：CPU>80%或Load>核数 → 自动限流保护

### 2.9 通知服务模块（商业级升级）

| 编号 | FR-NOTIFY-001 |
| 名称 | 告警通知与补发（商业级MQ+DB双保障） |
| 优先级 | P1 |
| 描述 | 告警触发后通过OpenFeign异步调用通知服务发送短信/邮件/电话/钉钉/企微/App推送等多通道通知。通知服务不可用时Sentinel熔断降级，FallbackFactory执行DB审计+MQ延迟重试双保障，配合XXL-Job分布式监控。 |

功能点：
1. **异步发送**：告警创建后通过@Async异步调Feign发送通知，不阻塞主流程
2. **多通道支持**：7通道枚举(SMS/EMAIL/APP_PUSH/PHONE/DINGTALK/WECOM/MANUAL)
3. **熔断降级**：通知服务异常率超过阈值时自动熔断30s，FallbackFactory提供DB+MQ双重保障
4. **DB审计**：通知失败时写入t_notification_retry表记录失败原因和上下文，用于审计追溯
5. **MQ延迟重试（核心升级）**：替代@Scheduled固定2分钟DB轮询，基于RabbitMQ DLX + Per-Message TTL：
   - 架构：DirectExchange → waitQueue(设TTL) → TTL过期 → DLX回流 → retryQueue → 消费者处理
   - 指数退避：1min→2min→4min→8min→16min→30min封顶，每次重试TTL翻倍
   - Per-Message TTL：每条消息独立设置过期时间，互不影响
   - confirmCallback：生产者发送确认回调，生产级可靠性
6. **通知重试消费者**：NotificationRetryConsumer消费retry.queue，执行：
   - 成功→ACK确认+DB UPDATE SENT
   - 失败→retryCount++：<5次重新入wait队列TTL加倍、≥3次通道降级、≥5次进DLQ
7. **通道降级策略**：SMS失败3次→自动切PHONE电话通知→PHONE也失败→MANUAL人工介入
8. **DLQ死信队列**：超过5次重试进入DLQ死信队列，人工介入排查
9. **XXL-Job分布式监控**：
   - notificationMonitorJob（每10分钟）：监控DLQ深度>10条触发告警、统计FAILED记录、计算成功率
   - notificationCleanupJob（每天2点）：清理7天前已SENT的通知记录

### 2.10 端云协同模块

| 编号 | FR-SYNC-001 |
|------|-------------|
| 名称 | 端云协同 |
| 优先级 | P1 |
| 描述 | 实现设备端与云端的数据同步、指令下发和配置推送，确保端云数据最终一致性。 |

**功能点：**

1. **指令下发**：平台通过 MQTT 向指定设备或设备分组下发指令（重启、配置更新、数据上报频率调整等），设备执行后回执结果。
2. **配置推送**：支持向设备批量推送配置参数，设备端接收后应用新配置并返回确认。
3. **数据同步策略**：
   - 实时同步：关键属性变更立即同步
   - 定时同步：非关键属性定时批量上报
   - 断线补传：设备离线期间本地缓存数据，上线后批量补传到平台
4. **设备时间同步**：平台定期向设备下发 NTP 时间同步指令，确保设备时钟与平台一致。

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|------|------|
| 设备接入并发 | 支持 10 万+ 设备同时在线 |
| 设备状态查询 | P99 响应时间 < 200ms（Redis 缓存命中） |
| MQTT 消息吞吐 | 峰值 10 万条/秒 |
| OTA 并发升级 | 支持 5000 台设备同时升级 |
| 告警延迟 | 从事件触发到告警生成 < 5 秒 |

### 3.2 可用性需求

| 指标 | 要求 |
|------|------|
| 系统可用性 | 99.9% |
| 数据持久性 | 告警、OTA 记录不丢失 |
| 故障恢复 | 单节点故障自动切换，恢复时间 < 30 秒 |

### 3.3 安全需求

- 设备认证采用 Token 签名机制（deviceId + deviceSecret + timestamp）
- MQTT Broker 启用 TLS 加密传输
- 管理后台接口需鉴权，敏感操作记录审计日志
- 固件包上传后 MD5 校验，设备端下载后校验完整性

**微服务划分：**

| 服务 | 职责 |
